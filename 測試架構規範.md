## 1. 設計原則與目標

### 1.1 核心目標

```
┌─────────────────────────────────────────────────────────────────┐
│                      測試架構目標權重                            │
├─────────────────────────────────────────────────────────────────┤
│  1. 降低開發成本 ████████████████████████████████████ 40%       │
│  2. 提高正確率   ██████████████████████████████ 30%             │
│  3. 提升開發速度 ██████████████████████ 20%                     │
│  4. 降低維護成本 ██████████████ 10%                             │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 設計原則

| 原則 | 說明 | 實踐方式 |
|:---|:---|:---|
| **宣告優於程式碼** | 能用註解/配置的，不寫程式碼 | `@QueryFilter`, Snapshot JSON |
| **結構化輸出** | 回傳物件而非基本型別 | `CalculationResult`, `QueryGroup` |
| **Service 只編排** | Service 不做決策，只做組裝 | 業務邏輯在 Domain 層 |
| **測試即文檔** | 快照 JSON 就是系統說明書 | `snapshots/*.json` |

### 1.3 傳統 vs 架構驅動測試

| 面向 | 傳統測試 | 架構驅動測試 |
|:---|:---|:---|
| 測試對象 | 程式碼怎麼跑 (How) | 產出的數據合約 (What) |
| Mock 使用 | 到處都是 Mock | 幾乎不 Mock，只攔截結果物件 |
| DB 依賴 | 沒資料庫就沒法測 | 核心邏輯完全脫離資料庫 |
| 維護反饋 | 「改 A 壞 B」 | 「精準報警」 |
| 測試代碼量 | 比業務代碼還長 | 3-5 行搞定 |

### 2.1 整體架構演進

├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  現有架構                        目標架構                        │
│  ─────────                       ─────────                       │
│                                                                 │
│  Service Factory Pattern    →    + 可攔截的結果物件              │
│  (execCommand/getResponse)       + Hook Points                  │
│                                                                 │
│  Fluent-Query-Engine        →    + QueryGroup 可序列化          │

### 2.3 Domain 層調整：結果物件化

**現有方式（回傳基本型別）：**

```java
public class Employee {
    public BigDecimal calculateBonus(Policy policy) {
        return new BigDecimal("50000");  // 只有數字，無法追蹤計算過程
    }
}
```

**調整後（回傳結構化結果）：**

```java
// 計算結果物件
public class CalculationResult<T> {
    private T finalValue;                    // 最終結果
    private List<String> appliedRules;       // 套用的規則清單
    private Map<String, Object> breakdown;   // 計算明細
    private LocalDateTime calculatedAt;      // 計算時間

    // Builder pattern...
}

// Domain Entity
public class Employee {
    public CalculationResult<BigDecimal> calculateBonus(Policy policy) {
        return CalculationResult.<BigDecimal>builder()
            .finalValue(new BigDecimal("50000"))
            .addRule("BASE_SALARY_RATE", "月薪 x 1.5")
            .addRule("PERFORMANCE_BONUS", "績效加成 10%")
            .breakdown("baseSalary", 40000)
            .breakdown("performanceBonus", 10000)
            .build();
    }
}
```

**測試效益：**

```java
@Test
void calculateBonus_ShouldMatchSnapshot() {
    CalculationResult<BigDecimal> result = employee.calculateBonus(policy);

    // 一行搞定：驗證結果 + 規則 + 明細
    FluentAssert.assertMatchesSnapshot("bonus_standard_case.json", result);
}
```

### 2.4 Service 層調整：可攔截設計

與 `BaseCommandService` / `BaseQueryService` 整合，新增可攔截點：

```java
public abstract class BaseQueryService<REQ, RES> implements QueryApiService<REQ, RES> {

    // 新增：讓測試可以攔截的 Hook
    protected QueryGroup lastQueryGroup;  // 儲存最後執行的查詢條件

    @Override
    public final RES getResponse(REQ request, JWTModel currentUser) throws Exception {
        // ... 現有邏輯 ...

        RES response = doQuery(request, currentUser);

        return response;
    }

    /**
     * 子類實作查詢邏輯時，應呼叫此方法記錄 QueryGroup
     */
    protected <T> Page<T> executeQuery(QueryGroup queryGroup, Pageable pageable,
                                        BaseRepository<T> repository) {
        this.lastQueryGroup = queryGroup;  // 記錄供測試攔截
        return repository.findPage(queryGroup, pageable);
    }

    // 測試用：取得最後執行的查詢條件
    public QueryGroup getLastQueryGroup() {
        return lastQueryGroup;
    }
}
```

## 3. 測試金字塔與分層策略

### 3.1 測試金字塔

```
                    ┌─────────┐
                    │   E2E   │  ← 少量：關鍵業務流程
                   ┌┴─────────┴┐
                   │Integration│  ← 中量：API 契約測試
                  ┌┴───────────┴┐
                  │    Unit     │  ← 大量：Domain + 快照測試
                 └──────────────┘
```

### 3.2 各層測試職責

| 層級 | 測試對象 | 方法 | 執行頻率 |
|:---|:---|:---|:---|
| **Unit** | Domain Entity 計算邏輯 | 快照比對 | 每次提交 |
| **Unit** | QueryGroup 組裝邏輯 | 快照比對 | 每次提交 |
| **Integration** | API → Service 契約 | MockMvc + 快照 | 每次提交 |
| **Integration** | QueryEngine → DB | H2 + 固定資料集 | 每日/PR |
| **E2E** | 關鍵業務流程 | Testcontainers | 發版前 |

## 4. 三階測試法

### 4.1 概述

```
┌─────────────────────────────────────────────────────────────────┐
│                    三階測試法                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  第一階：引擎契約測試 (Engine Contract Test)                     │
│  ─────────────────────────────────────────                      │
│  對象：UltimateQueryEngine                                      │
│  做法：H2 固定資料集，測試所有 Operator 與 Auto-Join            │
│  目的：確保「翻譯機」不會壞，後續不需跑 DB                       │
│                                                                 │
│  第二階：業務組裝測試 (Business Assembly Test)                   │
│  ─────────────────────────────────────────                      │
│  對象：Application Service                                      │
│  做法：攔截 QueryGroup，驗證條件組裝是否正確                    │
│  工具：QueryGroupAssert DSL 或 快照比對                         │
│                                                                 │
│  第三階：資料工廠自動化 (Data Factory Automation)                │
│  ─────────────────────────────────────────                      │
│  做法：DTO 直接生成 QueryGroup（透過 @QueryFilter）             │
│  目的：測試案例不再需要手動組裝條件                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 第一階：引擎契約測試

**目的**：一次性驗證 QueryEngine 的所有操作符，後續業務測試不再碰 DB。

```java
@SpringBootTest
@Testcontainers
class QueryEngineContractTest {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:15");

    @BeforeAll
    static void setupTestData() {
        // 載入固定測試資料集
        executeSqlScript("test-data/engine_contract_data.sql");
    }

    @ParameterizedTest
    @MethodSource("operatorTestCases")
    void operator_ShouldProduceCorrectResult(Operator op, Object value, int expectedCount) {
        QueryGroup query = QueryBuilder.create()
            .and("status", op, value)
            .build();

        List<Entity> result = repository.findAll(query);

        assertThat(result).hasSize(expectedCount);
    }

    static Stream<Arguments> operatorTestCases() {
        return Stream.of(
            Arguments.of(Operator.EQ, "ACTIVE", 10),
            Arguments.of(Operator.NE, "ACTIVE", 5),
            Arguments.of(Operator.LIKE, "John%", 3),
            Arguments.of(Operator.IN, List.of("A", "B"), 7),
            Arguments.of(Operator.IS_NULL, null, 2),
            Arguments.of(Operator.GTE, 100, 8)
            // ... 所有 Operator
        );
    }
}
```

### 4.3 第二階：業務組裝測試

**目的**：驗證 Service 組裝的 QueryGroup 是否符合業務意圖。

```java
@ExtendWith(MockitoExtension.class)
class EmployeeQueryServiceTest {

    @Mock
    private EmployeeRepository repository;

    @InjectMocks
    private GetEmployeeListServiceImpl service;

    @Test
    void searchEmployee_ShouldBuildCorrectQuery() {
        // 1. Arrange
        GetEmployeeListRequest req = new GetEmployeeListRequest();
        req.setDepartmentName("研發部");
        req.setStatus("ACTIVE");

        ArgumentCaptor<QueryGroup> captor = ArgumentCaptor.forClass(QueryGroup.class);

        // 2. Act
        service.getResponse(req, mockUser);

        // 3. Assert - 使用 DSL
        verify(repository).findPage(captor.capture(), any());

        QueryGroupAssert.assertThat(captor.getValue())
            .isJunction(LogicalOp.AND)
            .hasCondition("department.name", Operator.EQ, "研發部")
            .hasCondition("status", Operator.EQ, "ACTIVE");
    }

    @Test
    void searchEmployee_ShouldMatchSnapshot() {
        // 更簡潔：直接快照比對
        GetEmployeeListRequest req = new GetEmployeeListRequest();
        req.setDepartmentName("研發部");

        service.getResponse(req, mockUser);

        FluentAssert.assertMatchesSnapshot(
            "employee_search_by_dept.json",
            service.getLastQueryGroup()
        );
    }
}
```

### 4.4 第三階：資料工廠自動化

**目的**：利用 `@QueryFilter` 註解自動生成 QueryGroup，減少手動組裝。

```java
// DTO 定義
public class EmployeeSearchRequest {
    @QueryFilter(field = "name", op = Operator.LIKE)
    private String name;

    @QueryFilter(field = "department.name", op = Operator.EQ)
    private String departmentName;

    @QueryFilter(field = "status", op = Operator.IN)
    private List<String> statuses;
}

// 測試：自動從 DTO 生成 QueryGroup
@Test
void dto_ShouldAutoGenerateQueryGroup() {
    EmployeeSearchRequest req = new EmployeeSearchRequest();
    req.setName("John");
    req.setDepartmentName("IT");

    QueryGroup group = QueryBuilder.fromDto(req);

    // 驗證自動生成的結構
    QueryGroupAssert.assertThat(group)
        .hasCondition("name", Operator.LIKE, "John")
        .hasCondition("department.name", Operator.EQ, "IT");
}
```

### 6.2 BaseServiceTest

```java
/**
 * Service 層測試基類
 * 提供 QueryGroup 攔截與快照比對能力
 */
public abstract class BaseServiceTest<S> {

    @Mock
    protected BaseRepository<?> repository;

    protected ObjectMapper snapshotMapper = new QuerySnapshotMapper();

    /**
     * 驗證 Service 產出的 QueryGroup
     */
    protected void verifyQuery(Runnable serviceCall, String snapshotName) {
        ArgumentCaptor<QueryGroup> captor = ArgumentCaptor.forClass(QueryGroup.class);

        // 執行 Service
        serviceCall.run();

        // 攔截 QueryGroup
        verify(repository, atLeastOnce()).findPage(captor.capture(), any());
        QueryGroup actualGroup = captor.getValue();

        // 快照比對
        SnapshotUtils.compareOrUpdate(snapshotName, actualGroup, snapshotMapper);
    }

    /**
     * 驗證計算結果
     */
    protected <T> void verifyResult(T result, String snapshotName) {
        SnapshotUtils.compareOrUpdate(snapshotName, result, snapshotMapper);
    }
}
```

### 6.3 BaseApiContractTest

```java
/**
 * API 契約測試基類
 * 驗證 Request → Service 的參數轉換
 */
@SpringBootTest(webEnvironment = WebEnvironment.MOCK)
@AutoConfigureMockMvc
public abstract class BaseApiContractTest {

    @Autowired
    protected MockMvc mockMvc;

    @Autowired
    protected ObjectMapper objectMapper;

    @MockBean
    protected BaseRepository<?> repository;

    /**
     * 驗證 API 到 QueryGroup 的完整流程
     */
    protected void verifyApiToQuery(String url, Object request, String snapshotName)
            throws Exception {
        ArgumentCaptor<QueryGroup> queryCaptor = ArgumentCaptor.forClass(QueryGroup.class);

        // 執行 API
        mockMvc.perform(post(url)
                .contentType(MediaType.APPLICATION_JSON)
                .content(objectMapper.writeValueAsString(request)))
                .andExpect(status().isOk());

        // 攔截 QueryGroup
        verify(repository).findPage(queryCaptor.capture(), any());

        // 雙快照驗證
        SnapshotUtils.compareOrUpdate(snapshotName + "_dto", request);
        SnapshotUtils.compareOrUpdate(snapshotName + "_query", queryCaptor.getValue());
    }
}
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    TDD + 快照測試開發流程                        │

│                                                                 │
│  2. 寫測試 (先寫測試案例)                                       │
│     └── 定義快照檔名，執行測試 (預期失敗)                       │
│                                                                 │
│  3. 實作邏輯                                                    │
│     └── 開發 Service/Domain 邏輯                                │
│                                                                 │
│  4. 生成快照                                                    │
│     └── mvn test -DupdateSnapshots=true                        │
│                                                                 │
│  5. Review 快照                                                 │
│     └── 確認 JSON 內容符合預期                                  │
│                                                                 │
│  6. 提交                                                        │
│     └── git add snapshots/ && git commit                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 8.2 維護流程：欄位變更

當修改 Entity 欄位（例如 `dept` → `departmentCode`）：

1. **修改程式碼**
2. **執行測試** → 快照比對失敗，顯示 Diff
3. **確認是預期變更** → `mvn test -DupdateSnapshots=true`
4. **Review 更新後的快照** → 確認所有相關快照都正確更新
5. **提交變更**

## 9. 與現有架構整合

### 9.1 與 Service Factory Pattern 整合

現有的 `CommandApiService` / `QueryApiService` 需要小幅調整以支援測試攔截：

```java
// 現有介面不變
public interface QueryApiService<REQ, RES> {
    RES getResponse(REQ request, JWTModel currentUser) throws Exception;
}

// BaseQueryService 新增攔截點
public abstract class BaseQueryService<REQ, RES> implements QueryApiService<REQ, RES> {

    // 新增：儲存最後執行的查詢（測試用）
    @Getter
    private QueryGroup lastExecutedQuery;

    // 子類呼叫此方法執行查詢
    protected <T> Page<T> executePageQuery(QueryGroup query, Pageable pageable,
                                            BaseRepository<T> repo) {
        this.lastExecutedQuery = query;
        return repo.findPage(query, pageable);
    }
}
```

### 9.2 與 Generic-Library-Architecture 整合

`Generic-Library-Architecture.md` 中的泛型基類已經提供了良好的擴展點，測試基類可以直接整合：

```java
// 測試基類與泛型 Service 基類對應
public abstract class BaseCommandServiceTest<
    REQ,                              // Request 型別
    RES,                              // Response 型別
    S extends BaseCommandService<REQ, RES>  // Service 型別
> extends BaseServiceTest<S> {

    @InjectMocks
    protected S service;

    protected void verifyCommandResult(REQ request, String snapshotName) {
        RES result = service.doExecute(request, mockUser);
        SnapshotUtils.compareOrUpdate(snapshotName, result);
    }
}
```

### 9.3 與 Fluent-Query-Engine 整合

`QueryGroup` 需要實作可序列化介面：

```java
// QueryGroup.java 新增
@JsonSerialize(using = QueryGroupSerializer.class)
public class QueryGroup {
    // ... 現有程式碼 ...
}

// 自訂序列化器
public class QueryGroupSerializer extends JsonSerializer<QueryGroup> {
    @Override
    public void serialize(QueryGroup group, JsonGenerator gen,
                          SerializerProvider provider) throws IOException {
        gen.writeStartObject();
        gen.writeStringField("junction", group.getJunction().name());

        // 序列化條件
        gen.writeArrayFieldStart("conditions");
        for (FilterUnit unit : group.getConditions()) {
            gen.writeStartObject();
            gen.writeStringField("f", unit.getField());
            gen.writeStringField("op", unit.getOp().name());
            gen.writeObjectField("v", unit.getValue());
            gen.writeEndObject();
        }
        gen.writeEndArray();

        // 遞迴序列化子群組
        gen.writeArrayFieldStart("subGroups");
        for (QueryGroup sub : group.getSubGroups()) {
            serialize(sub, gen, provider);
        }
        gen.writeEndArray();

        gen.writeEndObject();
    }
}
```
