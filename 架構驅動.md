既然要打造一個「萬用套版」的**自動化測試基類**，我們的目標是讓開發者\*\*「只要寫業務邏輯，測試會自動找上門」\*\*。

這個基類必須封裝掉所有關於 Mockito 攔截、JSON 序列化比對、以及 `QueryGroup` 抓取的複雜細節。

## 專案骨架：`BaseArchitectureTest` 設計

這套設計的核心在於利用 Java 的**泛型（Generics）與反射（Reflection）**，讓子類測試只需要定義 DTO，剩下的比對自動完成。

### 1. 核心測試基類範例

```java
public abstract class BaseRepositoryTest<T, R extends BaseRepository<T>> {

    @Mock
    protected R repository; // 你的自研 Repository

    protected ObjectMapper mapper = new QuerySnapshotMapper(); // 剛才提到的精簡版 Mapper

    /**
     * 自動化驗證查詢合約
     * @param serviceCall 執行 Service 的 Lambda
     * @param snapshotName 快照檔名
     */
    protected void verifyQuery(Runnable serviceCall, String snapshotName) {
        // 1. 宣告攔截器
        ArgumentCaptor<QueryGroup> captor = ArgumentCaptor.forClass(QueryGroup.class);

        // 2. 執行業務邏輯 (例如呼叫 service.find...)
        serviceCall.run();

        // 3. 攔截並驗證
        verify(repository, atLeastOnce()).findPage(captor.capture(), any());
        QueryGroup actualGroup = captor.getValue();

        // 4. 自動化快照比對 (如果是學習模式就存檔，否則比對)
        SnapshotUtils.compareOrUpdate(snapshotName, actualGroup, mapper);
    }
}
```

### 2. 實際套版後的開發體驗

假設你要開發一個新的「訂單查詢」功能，你的測試代碼會簡化到令人髮指：

class OrderServiceTest extends BaseRepositoryTest<Order, OrderRepository> {

## 為什麼這套「基類套版」能大幅降低成本？

### 1. 規範強制化 (Architectural Guardrails)

開發者被強迫繼承 `BaseRepositoryTest`。如果他們不產生 `QueryGroup`，測試就跑不過。這確保了 **「架構驅動」** 不是口號，而是物理限制。

### 2. 測試即文檔 (Executable Documentation)

`src/test/resources/snapshots/` 下的 JSON 檔案就是最精準的系統說明書。新進同事不用看 SQL，看這些 JSON 就能理解業務規則。

### 3. AI 的完美燃料

當你要擴充功能時，你可以直接把 `order_vip_paid_search.json` 丟給 AI 並說：

> 「這是目前的查詢快照，請幫我修改 `OrderService`，增加一個『過期訂單排除』的邏輯，並產出新的快照預期值。」

## API 層自動化快照基類：`BaseApiContractTest`

這套設計利用 Spring 的 `MockMvc`，但不測資料庫，只測 **「輸入 Request -\> 轉化意圖」** 的過程。

### 1. 基類設計範例

```java
public abstract class BaseApiContractTest<C> {

    @Autowired
    protected MockMvc mockMvc;

    @Autowired
    protected ObjectMapper objectMapper;

    /**
     * 驗證 API 到 Service 的參數轉換契約
     * @param url API 路徑
     * @param requestPayload 傳入的 DTO 或 JSON
     * @param snapshotName 快照名稱
     */
    protected void verifyApiContract(String url, Object requestPayload, String snapshotName) throws Exception {
        // 1. 執行 API 呼叫
        String jsonRequest = objectMapper.writeValueAsString(requestPayload);
        
        mockMvc.perform(post(url)
                .contentType(MediaType.APPLICATION_JSON)
                .content(jsonRequest))
                .andExpect(status().isOk());

        // 2. 這裡我們可以攔截 Service 接收到的 DTO
        // 配合前面提到的 Service 攔截邏輯，將「最終生成的 DTO 狀態」存成快照
        SnapshotUtils.compareOrUpdate(snapshotName + "_dto", requestPayload, objectMapper);
    }
}
```

## 3. 為什麼這能徹底改變開發體驗？

### A. 解決「預設值」與「格式」的痛

很多 Bug 發生在：前端傳 `null`，後端 DTO 沒設好預設值；或是日期格式 `yyyy-MM-dd` 轉不進去。

  * **架構驅動：** 快照會立刻顯示轉換後的 DTO 狀態。如果轉換失敗，快照比對會直接報錯，你連 Debug 都不用打。

### B. 重構的「零恐懼」

當你想把 DTO 裡的 `course_name` 改名為 `name`：

1.  修改代碼。
2.  跑測試，快照報錯（顯示欄位名稱不符）。
3.  確認這是預期的改動，執行 `Update Snapshots`。
4.  **維護完成。**

### C. 文檔即代碼 (Swagger 的完美補充)

Swagger 告訴前端「可以傳什麼」，而你的 **Snapshot JSON** 告訴前端「你傳了這個，後端會怎麼解讀」。這對前後端協作有極大的信任提升。

### 1\. 核心實作：`BaseFlowTest` (意圖鏈結測試)

這套基類會自動完成以下流程：**模擬 API 呼叫 -\> 攔截 Service 產出的 QueryGroup -\> 生成雙重快照。**

```java
public abstract class BaseFlowTest {
    
    @Autowired protected MockMvc mockMvc;
    @Autowired protected ObjectMapper objectMapper;
    
    // 關鍵：攔截我們自研 Repository 的動作
    @MockBean protected BaseRepository<?> repository; 

    protected void verifyFlow(String url, Object request, String snapshotName) throws Exception {
        // 1. 準備攔截 QueryGroup
        ArgumentCaptor<QueryGroup> queryCaptor = ArgumentCaptor.forClass(QueryGroup.class);
        
        // 2. 模擬 API 請求
        mockMvc.perform(post(url)
                .contentType(MediaType.APPLICATION_JSON)
                .content(objectMapper.writeValueAsString(request)))
                .andExpect(status().isOk());

        // 3. 獲取 Service 最終生成的 QueryGroup
        verify(repository).findPage(queryCaptor.capture(), any());
        QueryGroup actualQuery = queryCaptor.getValue();

        // 4. 產出雙快照
        // 快照 A: 前端傳入後的 DTO 狀態 (驗證參數轉換、預設值)
        SnapshotUtils.compareOrUpdate(snapshotName + "_dto", request);
        // 快照 B: 最終生成的查詢邏輯 (驗證業務邏輯、權限過濾)
        SnapshotUtils.compareOrUpdate(snapshotName + "_query", actualQuery);
    }
}
```
