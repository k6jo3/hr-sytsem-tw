# 人力資源系統開發工作計畫與可行性評估報告

## 1. 執行摘要 (Executive Summary)
**審查狀態：** 目前的文件集（需求、架構、設計）**大致就緒**（Mostly Ready），可以用於開發。但在特定模組開始編碼之前，仍有幾個關鍵缺口需要解決。
- **系統架構：** ✅ 極佳。技術堆疊明確 (Spring Cloud/React)，模式清晰 (DDD/CQRS)，標準規範完整。
- **系統設計：** ⚠️ 高層設計良好，但針對複雜的勞動法規（勞基法）和稅務計算，缺乏「實作層級的邏輯細節」。
- **可行性：** 高。所選技術堆疊非常適合此類系統。

**建議：** 立即啟動 **第一階段（基礎建設）**。同時，指派資深 SA 或開發人員撰寫第二階段（考勤/薪資）複雜邏輯的「邏輯規格書」。

## 2. 問題與缺口分析 (Issues & Gap Analysis)
在進行全面開發之前，必須解決在文件審查中發現的以下問題：

### 2.1 關鍵阻礙 (必須在編碼特定模組前解決)
| 模組 | 問題 | 影響 | 建議行動 |
|:--- |:--- |:--- |:--- |
| **考勤管理** | **變形工時** 邏輯有提到但未定義。 | 開發人員無法在沒有確切數學規則（2/4/8週循環）的情況下撰寫計算引擎。 | 建立 `logic_spec/variable_hours_rules.md` (變形工時邏輯規格)，包含具體範例與邊界案例。 |
| **考勤管理** | **職災管理** 邏輯深度不足。 | 列出了「支援職災」，但缺少具體的法定補償計算公式。 | 建立 `logic_spec/occupational_injury_compensation.md` (職災補償邏輯規格)。 |
| **薪資管理** | **稅務與保險計算**。 | 勞保、健保及勞退級距的公式必須精確（需對應年份表）。 | 建立 `logic_spec/tax_insurance_tables_2025.md` (2025稅務保險級距表)。 |
| **IAM** | **SSO 整合細節**。 | 「支援 Google/Microsoft」對於帳號連結策略（自動建立？連結既有帳號？）描述模糊。 | 在 IAM 設計中定義帳號連結流程 (Account Linking Flow)。 |

## 3. 開發階段 (Development Phases)

### 第一階段：基礎建設與身份識別 (第 1-4 週)
**目標：** 建立平台骨架並啟用使用者登入/管理。
- **基礎設施：** 設定 Git, CI/CD, Monorepo 結構, API Gateway, Eureka, Config Server。
- **IAM 服務：** 實作登入 (JWT), 使用者管理, 角色/權限管理 (RBAC)。
- **組織服務：** 實作公司/部門結構, 員工檔案 (基礎版)。
- **前端：** 系統框架 (Shell app), 登入頁面, 儀表板版面配置, 選單系統。

### 第二階段：核心考勤與日常維運 (第 5-10 週)
**目標：** 追蹤時間與管理日常維運。 **(邏輯最重)**
- **考勤服務：**
    - 基礎打卡 (Web/GPS)。
    - 請假申請與簽核流程。
    - **關鍵：** 實作 *變形工時* 引擎（需先完成邏輯規格）。
- **簽核流程服務：** 實作處理簽核的引擎 (請假/加班)。
- **前端：** 考勤行事曆, 請假表單, 排班管理介面。

### 第三階段：薪資與薪酬 (第 11-16 週)
**目標：** 精確的薪資計算。
- **保險服務：** 管理勞/健保級距表。
- **薪資服務：**
    - 薪資結構定義 (津貼, 扣項)。
    - 計算引擎 (整合考勤數據)。
    - 薪資單產出 (PDF)。
- **前端：** 薪資儀表板 (管理員), 個人薪資單檢視 (員工)。

### 第四階段：專案與進階模組 (第 17-24 週)
**目標：** 專案管理與人才發展。
- **專案服務：** 專案/工項管理, 工時表 (Timesheets)。
- **績效/招募/訓練服務：** MVP 實作。
- **報表服務：** 用於高階分析的資料倉儲/ETL。
