# 📖 架構師交接手冊：合約驅動與全鏈路自動化體系

## 第一部分：核心架構哲學 (Core Philosophy)

1. **宣告式設計 (Declarative Design)**
* 為什麼我們不手寫 SQL：意圖與實作的分離。
* 從 `QueryGroup` 到動態翻譯引擎的運作原理。


2. **單一真理來源 (SSOT)**
* 需求文件、API 規格、前端 Mock、後端測試的四位一體。
* 為什麼 Markdown 是我們最好的協作媒介。



## 第二部分：技術實作與組件導覽 (Technical Components)

1. **後端：意圖引擎與快照測試**
* `BaseRepository` 的核心抽象與泛型設計。
* `SnapshotMatcher`：如何達成 0.1 秒級別的邏輯驗證。


2. **中台：合約判官引擎 (`ContractMatcher`)**
* 解析 Markdown 表格並轉換為 `QueryGroup` 斷言。
* 權限與角色過濾的自動注入機制。


3. **前端：基於合約的並行開發**
* 如何根據合約生成 Mock Data。
* 前後端數據類型與欄位名稱的強制對齊。



## 第三部分：角色協作 SOP (Standard Operating Procedures)

1. **SA 規範：如何撰寫合格的業務合約**
* 情境 ID (Scenario ID) 的命名原則。
* 業務條件的語義化表達 (Expression Syntax)。


2. **工程師規範：如何「滿足」合約**
* 攔截測試的標準寫法。
* 處理「快照不符」的偵錯流程。


3. **PM 規範：如何解讀進度報表**
* 從測試通過率看見真實的開發完成度。



## 第四部分：維護與演進 (Maintenance & Evolution)

1. **效能優化指南**
* 如何監控 `QueryGroup` 產出的 SQL 效能。
* 全局緩存策略的配置。


2. **技術債清理與引擎升級**
* 在不改動業務代碼的情況下，如何升級底層框架（如 JPA 升級到 R2DBC）。


3. **AI 輔助開發插件 (Optional)**
* 使用 Prompt 工程讓 AI 根據合約自動生成 DTO 與 Service 代碼。



---

### 架構師的平日守則 (The Architect's Creed)

這部分我會特別註記在手冊最後一頁，作為給繼任者的叮嚀：

> 「架構師的工作不是寫更多的代碼，而是**消除不必要的代碼**。如果你發現工程師開始在 Service 裡寫複雜的 `if-else`，那是架構的失職，請立刻重構底層引擎，將邏輯收回架構中。」
