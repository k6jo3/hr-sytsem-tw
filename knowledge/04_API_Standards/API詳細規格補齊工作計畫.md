**建立日期:** 2025-12-29
**最後更新:** 2025-12-29
**目的:** 追蹤各服務 API 詳細規格文件的補齊進度

---

## 1. 工作背景

### 1.1 問題說明

目前系統設計書中的 API 規格不夠完整，缺少以下關鍵資訊：
- API 的業務邏輯說明
- Request/Response 的完整欄位定義與驗證規則
- 錯誤碼與處理建議
- 領域事件發布說明

### 1.2 工作目標

為 14 個微服務建立完整的 API 詳細規格文件，每個 API 需包含：

| 項目 | 內容 |
|:---|:---|
| 基本資訊 | 端點、HTTP Method、Controller、Service、權限 |
| 用途說明 | 業務場景、使用者、解決問題 |
| 業務邏輯 | 驗證規則、處理步驟、計算邏輯、副作用 |
| Request 規格 | Headers、Path/Query Parameters、Request Body（含驗證規則） |
| Response 規格 | 成功回應結構、欄位說明、範例 JSON |
| 錯誤碼 | HTTP 狀態碼、錯誤碼、說明、處理建議 |
| 領域事件 | 事件名稱、Kafka Topic、Payload 結構 |

## 5. 工作流程

### 5.1 每個服務的工作步驟

```
1. 閱讀系統設計書
   ├── 理解服務定位與核心功能
   ├── 查看 API 總覽章節
   └── 了解領域模型與資料結構

2. 分析 API 清單
   ├── 列出所有 API 端點
   ├── 區分 Command (CUD) 與 Query (R)
   └── 確認 Controller 命名

3. 撰寫 API 詳細規格
   ├── 依照 API 文件標準模板格式
   ├── 補充業務邏輯說明
   ├── 定義完整 Request/Response
   ├── 列出錯誤碼
   └── 說明領域事件

4. 輸出文件
   └── 建立 spec/XX_服務系統設計書_API詳細規格.md
```

### 5.2 品質檢查項目

- [ ] 所有 API 端點都已涵蓋
- [ ] Controller/Service 命名符合規範
- [ ] 每個 API 都有完整的業務邏輯說明
- [ ] Request Body 包含所有欄位、類型、驗證規則
- [ ] Response 包含完整欄位說明與範例 JSON
- [ ] 錯誤碼遵循 `{CATEGORY}_{SPECIFIC_ERROR}` 命名
- [ ] 領域事件包含 Topic 與 Payload 結構
