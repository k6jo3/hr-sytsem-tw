# 技術決策紀錄 (Architecture Decision Records)

本文件記錄 HRMS 系統設計過程中的重大架構決策，說明每個決策的背景脈絡、選擇理由、被否決的替代方案，以及已知的代價與取捨。

> **格式說明：** 每筆 ADR 包含「背景 → 決策 → 理由 → 替代方案 → 代價」五段，確保未來維護者能理解「為什麼這樣做」。

---

### ADR-01: 採用微服務架構（Microservices Architecture）

**狀態：** 已採用 (Accepted)
**日期：** 2025-12-01

**背景：**

目標企業目前使用多套分散的獨立系統（考勤 ERP、薪資系統、招募平台各自獨立），員工資料需人工重複維護，且各系統之間幾乎沒有整合。新系統若採用單體架構，雖然初期開發快，但面臨以下問題：考勤模組上下班尖峰時段有高併發需求；薪資結算模組月底才執行但計算量極大；招募模組使用頻率則相對低頻。不同的擴展需求使得整體部署效率極低。

**決策：**

採用以 DDD Bounded Context 為邊界的微服務架構，將系統拆分為 14 個獨立服務，各自擁有獨立部署、獨立資料庫、獨立擴展能力。

**理由：**
- **獨立擴展**：考勤模組可在打卡尖峰時段水平擴展，不影響薪資或報表模組
- **故障隔離**：某服務宕機不會導致整個系統不可用（如：報表服務故障不影響打卡）
- **技術異質性**：未來可依需求為特定服務選用不同資料庫或語言
- **部署頻率分離**：薪資引擎變更無需重新部署整個系統

**替代方案（被否決）：**

| 方案 | 否決原因 |
|:---|:---|
| 單體架構 (Monolith) | 無法獨立擴展各模組；任一模組變更需全系統重新部署；長期維護成本高 |
| 模組化單體 (Modular Monolith) | 部署仍是整體；擴展靈活性不足；難以應對考勤打卡的突發尖峰流量 |

**代價與取捨：**
- 分散式系統複雜度顯著提升（服務發現、分散式追蹤、跨服務除錯）
- 需要額外的基礎設施（API Gateway、Kafka、Service Registry）
- 跨服務業務流程需設計最終一致性機制（SAGA）而非簡單資料庫交易

### ADR-02: 採用 Kafka 作為服務間非同步通訊骨幹

**狀態：** 已採用 (Accepted)
**日期：** 2025-12-03

**背景：**

微服務之間需要通訊。若採用同步 REST 呼叫，服務之間會產生強耦合：若 IAM 服務暫時不可用，ORG 服務建立員工就會失敗；若 PAY 服務回應慢，ATT 服務也會被拖慢。系統需要一種能讓服務**完全解耦、非同步驅動**的通訊機制，且在員工入職這類「一個事件觸發多個下游動作」的場景下，要能支援**一對多廣播**。

**決策：**

採用 Apache Kafka 作為 Domain Event 的發布/訂閱中介層。所有重大狀態變更（如 `EmployeeCreatedEvent`、`LeaveApprovedEvent`、`PayrollCalculatedEvent`）一律透過 Kafka 主題廣播，下游服務各自訂閱並獨立處理。

**理由：**
- **事件持久化**：Kafka 的 Log-based 儲存讓事件可以重播，新服務上線後可以追補歷史事件
- **高吞吐量**：適應考勤尖峰期每秒數百筆打卡紀錄的推播場景
- **消費者獨立性**：新增一個訂閱者不需要修改任何發布者程式碼
- **解耦容錯**：發布者服務與消費者服務可以獨立宕機，不會互相拖垮

**替代方案（被否決）：**

| 方案 | 否決原因 |
|:---|:---|
| 同步 REST 呼叫 | 服務強耦合；任何下游服務不可用即導致整個流程失敗 |
| RabbitMQ | 適合任務佇列，不支援事件重播；訊息確認後即刪除，無法讓新服務追補歷史 |
| gRPC 串流 | 適合低延遲即時串流，不適合此類業務事件廣播；長連線管理複雜 |

**代價與取捨：**
- 引入最終一致性：下游服務處理事件需要時間，不能保證即時強一致
- 事件 schema 版本管理需要設計（新增欄位需向後相容）
- 本地開發需要額外啟動 Kafka（用 Docker Compose 解決）

### ADR-03: 採用 DDD 四層架構（Domain-Driven Design）

**狀態：** 已採用 (Accepted)
**日期：** 2025-12-05

**背景：**

HRMS 系統的業務邏輯極為複雜——薪資計算涉及勞基法變形工時、請假類別折算、勞健保級距試算等多層規則；簽核流程涉及動態條件路由。若採用傳統三層架構（Controller → Service → DAO），業務邏輯容易被散佈在 Service 層的各種 if-else 中，難以測試、難以維護，且隨著需求增加快速腐化。

**決策：**

採用 DDD 四層架構：Interface Layer（介面層）→ Application Layer（應用層）→ Domain Layer（領域層）→ Infrastructure Layer（基礎設施層）。業務規則集中封裝在 Domain Layer 的 Aggregate Root 和 Domain Service 中，Application Layer 只做流程編排，不含業務決策。

**理由：**
- **業務規則集中**：薪資計算、請假驗證等複雜規則封裝在 Domain 物件內，不被 Spring 等框架污染
- **高可測試性**：Domain Layer 是純 Java POJO，無需啟動 Spring Context 即可單元測試
- **Bounded Context 邊界清晰**：每個服務的核心概念有明確邊界，避免概念洩漏
- **業務事件顯性化**：Domain Event（如 `LeaveApprovedEvent`）直接反映業務語言，不是技術產物

**替代方案（被否決）：**

| 方案 | 否決原因 |
|:---|:---|
| 傳統三層架構 (Controller-Service-DAO) | 業務邏輯散佈在 Service 層；複雜規則難以測試；貧血模型導致重複程式碼 |
| Transaction Script | 適合簡單 CRUD，無法應付薪資計算、SAGA 等複雜業務流程 |

**代價與取捨：**
- 學習曲線較高，團隊需理解 Aggregate Root、Value Object、Domain Event 等概念
- 初期建立 Domain Model 的時間成本較高
- 簡單的 CRUD 場景反而顯得過度設計（以 Adapter Pattern 緩解）

### ADR-04: 採用 CQRS 分離讀寫路徑

**狀態：** 已採用 (Accepted)
**日期：** 2025-12-05

**背景：**

HR 系統的讀寫比例嚴重不均衡：查詢操作（查員工列表、查薪資單、查考勤紀錄）遠多於寫入操作，且查詢通常需要跨多個關聯做複雜篩選和分頁。若讀寫共用同一路徑，複雜的查詢 SQL 會與業務異動邏輯混在同一個 Service 類別中，造成職責不清且效能優化困難。

**決策：**

採用 CQRS（Command Query Responsibility Segregation）：Controller 層分為 CmdController（POST/PUT/DELETE）和 QryController（GET）；Service 層分為 CommandApiService（含業務規則與事件發布）和 QueryApiService（純查詢，走 QueryDSL 宣告式引擎）。

**理由：**
- **讀寫路徑獨立優化**：Query 路徑可以直接用 QueryDSL 走 Read Model，不需要經過 Domain 層的驗證規則
- **職責清晰**：Command Service 負責業務決策，Query Service 負責資料投影，互不干擾
- **彈性擴展**：Query 路徑可獨立快取、分庫讀，不影響 Command 路徑
- **測試方便**：Query 邏輯可獨立測試過濾條件組裝（即合約測試的 Assembler 測試）

**替代方案（被否決）：**

| 方案 | 否決原因 |
|:---|:---|
| 讀寫共用 Service | 複雜查詢和業務邏輯混雜；Service 類別膨脹；效能難以獨立優化 |
| 完整 Event Sourcing | 實作複雜度極高；查詢需重播事件流；過度設計超出本系統需求 |

**代價與取捨：**
- Controller 和 Service 類別數量增加（每個資源有兩支 Controller）
- 需要維護 Command Model 和 Query Model 的同步（用 Kafka 事件驅動 Read Model 更新）

### ADR-05: 採用 Database-per-Service（每服務獨立資料庫）

**狀態：** 已採用 (Accepted)
**日期：** 2025-12-05

**背景：**

如果所有微服務共享同一個資料庫，表面上降低了複雜度，但實際上服務之間會透過 JOIN 查詢產生隱性耦合。一個服務的 Schema 變更可能破壞其他服務的查詢，部署不再獨立，資料庫也成為單點瓶頸。

**決策：**

每個微服務擁有自己獨立的 PostgreSQL Schema（或獨立 Database）。服務之間禁止跨庫 JOIN，所有跨服務的資料需求一律透過 API 呼叫或訂閱 Kafka 事件後在本地維護一份必要的快取副本（Anti-Corruption Layer）。

**理由：**
- **真正的服務獨立性**：Schema 變更不影響其他服務
- **故障隔離**：某服務的資料庫效能問題不會拖垮其他服務
- **資料模型自主**：每個服務可依自己的需求選擇最合適的資料結構

**替代方案（被否決）：**

| 方案 | 否決原因 |
|:---|:---|
| 共享資料庫 (Shared Database) | 服務之間隱性耦合；Schema 變更風險高；資料庫成為部署瓶頸 |
| 共享 Schema、但限制 JOIN | 實際上很難強制執行；隨著時間仍會出現跨服務 JOIN |

**代價與取捨：**
- 跨服務查詢需要 API Composition 或事件驅動的本地快取，複雜度增加
- 最終一致性而非強一致性
- 報表類需求（跨服務彙總）需要專門的 Reporting Service（RPT）維護 Read Model

### ADR-06: 採用 SAGA Pattern 處理薪資結算分散式交易

**狀態：** 已採用 (Accepted)
**日期：** 2025-12-08

**背景：**

薪資月結業務需要協調 ATT（考勤資料）、INS（保費計算）、PAY（最終薪資計算與鎖定）三個服務的資料，且這個流程必須是全成功或全補償——不能出現「考勤資料讀了但薪資沒算完」的中間污染狀態。傳統資料庫的 2PC（Two-Phase Commit）在微服務架構中不適用，因為跨服務沒有統一的事務協調器。

**決策：**

採用 Choreography-based SAGA Pattern：薪資結算流程被分解為一系列有**補償交易（Compensating Transaction）**的步驟，透過 Kafka 事件串連。每一步成功時發布下一步的觸發事件；失敗時發布補償事件，逆向回滾已執行的步驟。

**理由：**
- **適合微服務**：不需要跨服務的全域鎖，每個服務只鎖自己的資料
- **可追溯**：每個步驟的成功/失敗都有對應的 Kafka 事件，可稽核
- **彈性補償**：各步驟的補償邏輯各自實作，不依賴中央協調器

**替代方案（被否決）：**

| 方案 | 否決原因 |
|:---|:---|
| 2PC（Two-Phase Commit）| 微服務間無法使用分散式鎖；阻塞性強、效能差；任一服務宕機導致全程阻塞 |
| Orchestration SAGA | 需要一個中央 Saga Orchestrator 服務，增加單點風險；本系統用 Choreography 已足夠 |
| 不處理（接受部分失敗）| 薪資計算資料不一致是無法接受的業務風險 |

**代價與取捨：**
- 補償邏輯設計複雜，需要為每個步驟設計對應的 Undo 操作
- 需要 Idempotent（冪等性）設計，防止事件重複消費導致重複補償
- 除錯困難：失敗追蹤需要完整的分散式追蹤（Sleuth + Zipkin）

### ADR-07: 採用 Markdown 合約驅動測試框架（自研）

**狀態：** 已採用 (Accepted)
**日期：** 2026-01-15

**背景：**

傳統整合測試由工程師自行撰寫，測試內容反映的是「工程師對需求的理解」，而非「SA 定義的業務規格」。測試通過不等於功能符合業務需求，且需求文件和測試程式碼是兩個完全獨立的產出，隨時間容易脫節。需要一種機制讓「業務規格」直接成為「可執行的測試斷言」。

**決策：**

設計 Markdown Contract-Driven Testing Framework：SA 在 `contracts/{service}_contracts.md` 以 Markdown 表格定義業務場景（輸入參數、必須包含的查詢過濾條件、回應結構要求）；`MarkdownContractEngine` 解析這份規格；`BaseContractTest` 提供 `assertContract()` 方法，讓測試程式碼直接載入合約並做三層斷言：查詢條件正確性、回應欄位完整性、領域事件正確發布。

**理由：**
- **需求即測試**：SA 的 Markdown 規格直接成為測試的 ground truth，需求和測試不可能脫節
- **非對稱驗證**：工程師不能「自己測試自己」，必須對外部定義的規格負責
- **低門檻**：SA 只需會寫 Markdown 表格，不需要了解測試程式碼
- **三層驗證**：同時驗證查詢條件組裝、回應結構、副作用（Domain Event），比只測 API 回傳格式更全面

**替代方案（被否決）：**

| 方案 | 否決原因 |
|:---|:---|
| Pact（Consumer-Driven Contract）| 側重服務間 API schema 相容性，不驗證業務規則正確性；SA 無法直接定義 |
| 純整合測試 | 測試由工程師自行定義，仍是「自己測試自己」；與需求文件脫節 |
| OpenAPI + Contract Test | 只驗證 API 格式，不驗證業務邏輯（如：查詢條件是否包含軟刪除過濾）|

**代價與取捨：**
- 合約框架需要自行維護（不是開源工具）
- 合約文件格式有學習成本（SA 需要了解如何描述過濾條件）
- 目前以 Query 場景為主，Command 場景的三層驗證相對複雜

### ADR-08: 採用 QueryDSL 作為新功能的查詢引擎

**狀態：** 已採用 (Accepted)
**日期：** 2025-12-10

**背景：**

系統既有 MyBatis 作為持久層框架，但新功能的查詢條件越來越複雜（多維度篩選、動態條件組合、分頁排序）。MyBatis 的 XML 動態 SQL 需要手寫大量 `<if>` 判斷，可讀性差且難以維護。需要一個更具表達力的宣告式查詢方式，同時保留型別安全。

**決策：**

新功能的 Repository 和 Assembler 層採用 QueryDSL + JPA，搭配自研的 Fluent Query Engine（`QueryBuilder`）。`QueryBuilder` 提供宣告式的過濾條件 DSL，Assembler 類別負責將請求參數轉換成 `QueryGroup`，Repository 透過 `findPage(queryGroup, pageable)` 統一執行。既有功能保留 MyBatis 不強制遷移。

**理由：**
- **型別安全**：QueryDSL 透過 QClass 在編譯期檢查欄位名稱，避免 MyBatis 字串拼接的執行期錯誤
- **宣告式可讀**：過濾條件以物件表達，Assembler 的邏輯一目瞭然
- **可測試性**：`QueryGroup` 是結構化物件，合約測試可以直接斷言過濾條件，不依賴 SQL 字串比對
- **漸進遷移**：與 MyBatis 並存，不需要一次性重寫

**替代方案（被否決）：**

| 方案 | 否決原因 |
|:---|:---|
| 繼續用 MyBatis XML | 動態 SQL 的 `<if>` 區塊可讀性差；無法做結構化的過濾條件斷言測試 |
| Spring Data JPA（純 Specification）| Specification 的組合語法冗長；複雜查詢可讀性不如 QueryDSL |
| JOOQ | 授權費用（商業版）；與現有 JPA 生態系整合複雜度高 |

**代價與取捨：**
- 需要額外的 APT（Annotation Processing Tool）生成 QClass，增加建置時間
- 開發者需要熟悉 QueryDSL API
- MyBatis 和 QueryDSL 並存的雙軌策略需要統一規範，避免新功能混用
