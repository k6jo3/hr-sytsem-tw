# 核心業務循序圖 (Core Sequence Diagrams)

本文件匯整了 HRMS 系統中最核心的系統循序圖 (Sequence Diagram)。透過這些循序圖，展示了本專案在**事件驅動架構 (Event-Driven Architecture)** 下的運作機制，說明如何解決傳統單體架構的強耦合問題，並達成微服務間的非同步協作與單點故障隔離。

## 一、 新進員工入職流程 (Employee Onboarding Flow)

此流程為展示微服務特性的經典場景。當人資 (HR) 建立一筆新員工資料時，系統**不會**讓 Organization (組織模組) 依序同步呼叫 IAM (權限)、Payroll (薪資)、Insurance (保險) 等服務，而是採取「發布 / 訂閱 (Pub/Sub)」模式進行非同步處理，大幅縮短了寫入資料的等待時間，並提高系統容錯率。

```mermaid
sequenceDiagram

    %% 定義參與者
    actor HR as 人資/管理員(UI)
    participant ORG as ORG 微服務<br/>(Organization)
    participant DB_ORG as ORG 資料庫
    participant Kafka as Kafka<br/>(Event Bus)
    participant IAM as IAM 微服務<br/>(身份認證)
    participant INS as INS 微服務<br/>(勞健保)
    participant PAY as PAY 微服務<br/>(薪資)
    participant NTF as NTF 微服務<br/>(通知服務)

    %% 同步建立流程
    rect rgb(238, 242, 255)
        Note over HR, DB_ORG: Phase 1: 同步寫入與事件發布 (Synchronous)
        HR->>ORG: POST /api/v1/employees<br/>(新增員工資料)
        ORG->>ORG: 驗證必填欄位與格式 (Business Validation)
        ORG->>DB_ORG: INSERT 員工基本資料 (State Change)
        DB_ORG-->>ORG: 寫入成功 (Return ID)
        
        %% 發送事件
        ORG->>Kafka: Publish: EmployeeCreatedEvent<br/>{empId, name, email, deptId}
        Note right of ORG: 遵循 Outbox Pattern 或直接發送，確保事件不遺失。
        
        ORG-->>HR: HTTP 200 OK (建檔完成)
        Note left of ORG: HR 無須等待後續流程，<br/>前端畫面瞬間收到成功回饋！
    end

    %% 非同步訂閱流程
    rect rgb(240, 253, 244)
        Note over Kafka, NTF: Phase 2: 非同步業務處理 (Asynchronous Pub/Sub)

            Kafka-->>IAM: Consume: EmployeeCreatedEvent
            IAM->>IAM: 產生系統預設密碼與綁定基礎 Role
            IAM->>IAM: 寫入 IAM Schema (帳號就緒)
            
        and 保險投保
            Kafka-->>INS: Consume: EmployeeCreatedEvent
            INS->>INS: 產生預設勞健保級距檔案
            INS->>INS: 寫入 INS Schema (待 HR 呈報)
            
        and 薪資建檔
            Kafka-->>PAY: Consume: EmployeeCreatedEvent
            PAY->>PAY: 產生空白計薪結構與預設薪資帳戶格式
            PAY->>PAY: 寫入 PAY Schema
            
        and 發送歡迎信
            Kafka-->>NTF: Consume: EmployeeCreatedEvent
            NTF->>NTF: 渲染 Email 版面
            NTF->>HR (電子信箱): 寄送「歡迎入職與登入密碼信」
        end
    end
```

### 流程設計要點 (Architecture Design Points)

此設計涵蓋了以下三個重要技術決策：

1. **避免微服務間同步呼叫（解決系統耦合）**
   如果使用同步呼叫 (Synchronous REST Call)，當 IAM 或 Payroll 某一個服務無回應時，ORG 模組的新增操作就會跟著失敗並回滾。此外，未來若需增加新模組（例如資產管理模組發配電腦），ORG 的程式碼勢必面臨修改。
   改用 Kafka 事件驅動後，ORG 僅負責『廣播』員工建檔完成，後續處理不影響 ORG 的主流程，完全符合**單一職責原則 (SRP)** 與**開閉原則 (OCP)**。
   
2. **解決高併發下的效能瓶頸（提升使用者體驗）**
   建立帳號、配置保險、計算薪資結構乃至寄送 Email，整體流程可能耗時 3 至 5 秒。採用非同步處理後，ORG 寫入資料與將 Event 放進 Kafka 只需不到 50 毫秒 (ms)，使用端可瞬間收到成功回饋。其餘業務邏輯交由背景服務獨立且平行地處理，達到**最終一致性 (Eventual Consistency)**。

3. **分散式交易的事件不遺失保證（資料一致性）**
   為了避免資料庫寫入成功，但 Kafka 服務異常導致事件漏發。在架構設計上採用 **Outbox Pattern (發件箱模式)**，將領域事件先寫入關聯式資料庫的 `outbox_events` 表，與業務操作同屬一個 Transaction，再由系統排程或 Debezium (CDC) 非同步將 Event 推送至 Kafka，以達成 **At-Least-Once (至少一次)** 投遞保證。

## 二、 請假簽核流程 (Leave Approval Flow)

此流程展示 **WFL（工作流引擎）** 與 **ATT（考勤模組）** 之間透過 Kafka 解耦的多階簽核設計。員工提交請假申請後，系統立即回應成功，並非同步觸發簽核任務指派與主管通知，最終依審核結果更新假期餘額。

```mermaid
%%{init: {'theme': 'base', 'themeVariables': { 'fontSize': '16px', 'fontFamily': 'sans-serif', 'textColor': '#111111', 'actorTextColor': '#111111', 'messageTextColor': '#111111', 'noteTextColor': '#111111', 'lineColor': '#333333'}}}%%
sequenceDiagram
    autonumber

    actor Emp as 員工(UI)
    participant ATT as ATT 微服務<br/>(考勤)
    participant DB_ATT as ATT 資料庫
    participant Kafka as Kafka<br/>(Event Bus)
    participant WFL as WFL 微服務<br/>(工作流引擎)
    participant NTF as NTF 微服務<br/>(通知)
    actor Mgr as 直屬主管(UI)

    rect rgb(238, 242, 255)
        Note over Emp, DB_ATT: Phase 1: 提交申請 & 暫凍餘額 (Synchronous)
        Emp->>ATT: POST /api/v1/leaves<br/>{type: 特休, startDate, endDate}
        ATT->>ATT: 驗證：餘額充足？時段不衝突？
        ATT->>DB_ATT: INSERT leave_requests (status: PENDING_APPROVAL)<br/>UPDATE leave_balances (暫凍對應時數)
        DB_ATT-->>ATT: 寫入成功
        ATT->>Kafka: Publish: LeaveRequestCreatedEvent<br/>{requestId, empId, type, hours, managerId}
        ATT-->>Emp: HTTP 200 OK (申請已送出，等待主管審核)
    end

    rect rgb(240, 253, 244)
        Note over Kafka, Mgr: Phase 2: 簽核任務指派 (Asynchronous)
        par 建立簽核任務
            Kafka-->>WFL: Consume: LeaveRequestCreatedEvent
            WFL->>WFL: 依流程定義建立 WorkflowInstance<br/>指派 ApprovalTask 給 managerId
        and 通知主管
            Kafka-->>NTF: Consume: LeaveRequestCreatedEvent
            NTF-->>Mgr: 推送簽核通知（站內信 + Email）
        end
    end

    rect rgb(255, 248, 230)
        Note over Mgr, NTF: Phase 3: 主管審核 (Decision)
        Mgr->>WFL: PUT /api/v1/workflow/tasks/{taskId}/approve<br/>{decision: APPROVED, comment: "同意"}

        alt 核准 (Approved)
            WFL->>Kafka: Publish: LeaveApprovedEvent<br/>{requestId, empId, hours}
            Kafka-->>ATT: Consume: LeaveApprovedEvent
            ATT->>DB_ATT: UPDATE leave_requests (status: APPROVED)<br/>確認扣除假期餘額（暫凍轉正式扣除）
            Kafka-->>NTF: Consume: LeaveApprovedEvent
            NTF-->>Emp: 通知員工：請假已核准
        else 駁回 (Rejected)
            WFL->>Kafka: Publish: LeaveRejectedEvent<br/>{requestId, empId, hours, reason}
            Kafka-->>ATT: Consume: LeaveRejectedEvent
            ATT->>DB_ATT: UPDATE leave_requests (status: REJECTED)<br/>還原暫凍的假期餘額
            Kafka-->>NTF: Consume: LeaveRejectedEvent
            NTF-->>Emp: 通知員工：請假已駁回（含原因）
        end
    end
```

### 流程設計要點

1. **ATT 與 WFL 完全解耦**：ATT 不直接呼叫 WFL，只需發布事件。未來若簽核規則改變（如增加 HR 複核），只需修改 WFL 的流程定義，ATT 零改動。

2. **餘額暫凍機制**：申請提交後立即凍結對應時數，防止同一時段被重複申請；駁回後自動解凍，不需人工介入。

3. **雙向事件確認**：核准和駁回各有獨立的 Domain Event，下游服務（ATT、NTF）各自訂閱並處理，流程可稽核、可重播。

## 三、 薪資月結 SAGA 補償流程 (Payroll Monthly Run with SAGA)

此流程展示 **SAGA Pattern** 如何在不使用分散式鎖（2PC）的前提下，確保薪資結算的資料一致性。薪資計算需跨 ATT、INS、PAY 三個服務聚合資料，任何步驟失敗都會觸發補償交易，不留中間污染狀態。

```mermaid
%%{init: {'theme': 'base', 'themeVariables': { 'fontSize': '16px', 'fontFamily': 'sans-serif', 'textColor': '#111111', 'actorTextColor': '#111111', 'messageTextColor': '#111111', 'noteTextColor': '#111111', 'lineColor': '#333333'}}}%%
sequenceDiagram
    autonumber

    actor HR as 人資管理員(UI)
    participant PAY as PAY 微服務<br/>(薪資結算)
    participant DB_PAY as PAY 資料庫
    participant ATT as ATT 微服務<br/>(考勤)
    participant INS as INS 微服務<br/>(勞健保)
    participant Kafka as Kafka<br/>(Event Bus)
    participant NTF as NTF 微服務<br/>(通知)

    rect rgb(238, 242, 255)
        Note over HR, INS: SAGA Step 1~3：啟動結算 & 跨服務資料聚合
        HR->>PAY: POST /api/v1/payroll/runs<br/>{year: 2026, month: 2}
        PAY->>DB_PAY: INSERT payroll_runs (status: PROCESSING)
        DB_PAY-->>PAY: OK (runId: R2026-02)
        PAY-->>HR: HTTP 202 Accepted (結算進行中)

        Note over PAY, INS: SAGA Step 2: API Composition 聚合各服務資料
        PAY->>ATT: GET /api/v1/attendance/monthly-summary?month=2026-02
        ATT-->>PAY: 全員考勤摘要（出勤天數、加班時數、請假扣薪）
        PAY->>INS: GET /api/v1/insurance/monthly-contributions?month=2026-02
        INS-->>PAY: 全員保費自付額清單（勞保 + 健保 + 勞退）
    end

    rect rgb(240, 253, 244)
        Note over PAY, NTF: SAGA Step 4~6：計算、鎖定與完成

        PAY->>PAY: 依序計算每位員工淨薪<br/>(底薪 + 加給 + 加班費 - 請假扣薪 - 保費 - 預扣稅)

        alt SAGA 成功路徑 (All Steps Pass)
            PAY->>DB_PAY: UPDATE payroll_runs (status: LOCKED)<br/>BULK INSERT payroll_items（每位員工明細）
            PAY->>PAY: 產出銀行轉帳媒體檔 (TXT/CSV)
            PAY->>Kafka: Publish: PayrollCompletedEvent<br/>{runId, month, totalNetPay}
            Kafka-->>NTF: Consume: PayrollCompletedEvent
            NTF->>NTF: 產生個人電子薪資單 PDF
            NTF-->>NTF: 寄送薪資單通知給全體員工
            PAY-->>HR: 結算完成（含媒體檔下載連結）
        else SAGA 補償路徑 (Compensating Transaction)
            Note over PAY: 計算失敗（如：某員工薪資結構異常）
            PAY->>DB_PAY: UPDATE payroll_runs (status: FAILED)<br/>DELETE 本次已寫入的中間計算資料（補償回滾）
            PAY->>Kafka: Publish: PayrollFailedEvent<br/>{runId, failedStep, errorDetail}
            Kafka-->>NTF: Consume: PayrollFailedEvent
            NTF-->>HR: 緊急警示通知（結算失敗 + 錯誤明細）
            Note over PAY: 資料庫回到結算前的乾淨狀態<br/>HR 修正錯誤後可重新觸發結算
        end
    end
```

### 流程設計要點

1. **為何選 SAGA 而非 2PC**：跨微服務無法使用全域資料庫鎖。SAGA 的每個步驟只鎖自己的資料，失敗時透過補償交易逆向回滾，不阻塞其他服務。

2. **API Composition vs 事件驅動**：薪資結算需要「同步聚合」考勤與保費資料（當下最新數據），因此採用同步 REST API Composition，而非非同步事件訂閱。

3. **冪等性設計**：`payroll_runs` 以 `(year, month)` 為唯一鍵，重複觸發不會產生重複批次；補償交易使用 `DELETE WHERE run_id = ?` 確保冪等清除。

## 四、 員工離職流程 (Employee Offboarding Flow)

此流程為入職流程的鏡像，同樣展示事件驅動的**一事件多服務響應**模式。HR 只需執行一個「辦理離職」操作，後續的帳號停用、退保、最終薪資結算全部由各服務自主完成，無需人工逐一操作。

```mermaid

sequenceDiagram
    autonumber

    actor HR as 人資管理員(UI)
    participant ORG as ORG 微服務<br/>(組織員工)
    participant DB_ORG as ORG 資料庫
    participant Kafka as Kafka<br/>(Event Bus)
    participant IAM as IAM 微服務<br/>(身份認證)
    participant INS as INS 微服務<br/>(勞健保)
    participant PAY as PAY 微服務<br/>(薪資)
    participant NTF as NTF 微服務<br/>(通知)

    rect rgb(238, 242, 255)
        Note over HR, DB_ORG: Phase 1: 離職建檔 & 事件發布 (Synchronous)
        HR->>ORG: POST /api/v1/employees/{id}/offboard<br/>{terminationDate, reason, handoverTo}
        ORG->>ORG: 前置檢查：確認無未結清的代墊款、<br/>無進行中的待審核表單
        ORG->>DB_ORG: UPDATE employees (status: TERMINATED, terminationDate)<br/>INSERT offboarding_records (交接人、離職原因)
        DB_ORG-->>ORG: 寫入成功
        ORG->>Kafka: Publish: EmployeeTerminatedEvent<br/>{empId, terminationDate, remainingLeaveDays}
        ORG-->>HR: HTTP 200 OK (離職手續完成)
        Note left of ORG: HR 無須等待帳號停用、<br/>退保等後續流程即可完成操作
    end

    rect rgb(253, 237, 237)
        Note over Kafka, NTF: Phase 2: 非同步自動化離職作業 (Asynchronous Pub/Sub)

        par 帳號排程停用
            Kafka-->>IAM: Consume: EmployeeTerminatedEvent
            IAM->>IAM: 設定帳號停用排程<br/>(生效時間 = terminationDate 23:59)
            Note right of IAM: 允許員工在離職日當天<br/>仍可正常使用系統
        and 勞健保退保
            Kafka-->>INS: Consume: EmployeeTerminatedEvent
            INS->>INS: 建立退保申請記錄<br/>(退保日 = terminationDate)
            INS->>INS: 計算最後一個月保費（按比例）
        and 最終薪資結算
            Kafka-->>PAY: Consume: EmployeeTerminatedEvent
            PAY->>PAY: 計算未休特休折現金額<br/>(remainingLeaveDays × 日薪)
            PAY->>PAY: 標記該員工於下次月結時<br/>納入離職月份最終薪資計算
        and 寄送離職通知
            Kafka-->>NTF: Consume: EmployeeTerminatedEvent
            NTF-->>HR: 離職辦理完成確認單（含交接清單）
            NTF-->>NTF: 寄送離職感謝信給員工
        end
    end
```

### 流程設計要點

1. **與入職流程的對稱設計**：`EmployeeTerminatedEvent` 與 `EmployeeCreatedEvent` 結構對稱，各服務訂閱自己關心的事件並自主處理，ORG 不需要知道有哪些下游服務存在。

2. **帳號停用的時間差設計**：IAM 設定的是**排程停用**而非立即停用，允許員工在離職日當天完成最後的工作交接（如最後一次系統操作），避免即時停用造成業務中斷。

3. **未休特休折現計算**：PAY 在收到事件時先計算折現金額並標記，等到當月薪資結算時一併納入最終薪資，確保計算時機與薪資週期對齊，不造成額外的帳款。
