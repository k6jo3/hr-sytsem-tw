# 核心業務流程圖 (Business Flowcharts & Swimlane)

本文件展示系統中跨模組、高複雜度的核心業務邏輯流程。透過泳道圖 (Swimlane Diagram) 清晰界定出不同角色 (Actors) 以及後端微服務集群 (Microservices) 於各個階段應負擔的責任與系統邊界條件。

---

## 一、 考勤結算至薪資發放流程 (Attendance to Payroll Flow)

此流程涵蓋了企業人資系統中最關鍵的「月結處理」。它不僅牽涉前端使用者的互動，更涉及【考勤模組 (ATT)】與【薪資模組 (PAY)】之間的跨服務資料同步與非同步事件運算。

> **圖表格式：** PlantUML 渲染｜原始碼：`04_核心業務流程圖.puml`｜渲染指令：`java -jar tools/plantuml.jar -smetana -o diagrams 04_核心業務流程圖.puml`

![考勤結算至薪資發放流程](diagrams/04_考勤結算至薪資發放流程.png)

### 流程設計要點 (Architecture Design Points)

此設計展示了複雜的分散式結算邏輯如何被穩健地處理：

1. **防呆與依賴檢查 (Constraint & Validation)**：在進入薪資計算（Phase 3）前，必須確保考勤資料已經完全鎖定（Phase 2）。系統防呆設計會阻擋在仍有請假單卡在主管端的狀態下進行結算，確保資料一致性 (Data Integrity)。
2. **事件與 API Query 的混合應用 (Hybrid Communication)**：
   * 考勤結班後會發出 Event 告知 PAY 模組「這段期間的資料已鎖定」。
   * 但計薪時，基於資料即時性且為了避免把龐大運算參數塞入 Kafka 訊息中，PAY 模組是主動呼叫 ATT 與 INS 的 Query API 索取精確的時數與級距數字（API Composition 模式）。此做法降低了 Queue 的負載並提高了資料獲取的正確性。

## 二、員工入職流程 (Employee Onboarding Flow)

此流程描述從 HR 建立員工資料到完成所有系統初始化的跨服務協作。涉及【組織模組 (ORG)】、【IAM 模組】、【保險模組 (INS)】、【薪資模組 (PAY)】四個服務的非同步事件串接。

```plantuml
@startuml Employee Onboarding Flow
|HR 人員|
start
:開啟「新增員工」頁面;
:填寫基本資料
（姓名、身分證、部門、職稱、到職日）;
:送出新增請求;

|HR02 Organization Service|
:驗證資料完整性
（身分證格式、部門存在性）;
:建立 Employee Aggregate;
:儲存至 DB;
:發布 **EmployeeCreatedEvent**
（employee_id, name, department_id, hire_date）;

fork
  |HR01 IAM Service|
  :訂閱 EmployeeCreatedEvent;
  :自動建立使用者帳號
  （username = 員工編號, 預設密碼）;
  :指派 EMPLOYEE 角色;
  :發布 **UserAccountCreatedEvent**;
fork again
  |HR05 Insurance Service|
  :訂閱 EmployeeCreatedEvent;
  :建立勞保投保記錄
  （依投保級距自動計算）;
  :建立健保投保記錄;
  :發布 **InsuranceEnrolledEvent**;
fork again
  |HR04 Payroll Service|
  :訂閱 EmployeeCreatedEvent;
  :建立薪資結構
  （底薪、津貼、扣項）;
  :關聯薪資項目至員工;
  :發布 **SalaryStructureCreatedEvent**;
end fork

|HR 人員|
:收到系統通知「員工入職流程完成」;
:確認各模組資料正確;
stop

note right
  **跨服務通訊模式：**
  Event-Driven（Kafka）

  **最終一致性保證：**
  每個訂閱者獨立處理，
  失敗時進入 DLQ 重試
end note
@enduml
```

### 流程設計要點

1. **事件驅動解耦（Event-Driven Decoupling）**：HR02 只負責建立員工本身，其餘服務透過訂閱 `EmployeeCreatedEvent` 自動完成後續作業，實現服務間零耦合。
2. **最終一致性（Eventual Consistency）**：三個下游服務（IAM、Insurance、Payroll）平行處理，任一服務失敗不影響其他服務，透過 Dead Letter Queue 機制確保最終一致。
3. **冪等性設計（Idempotency）**：每個訂閱者以 `employee_id` 作為冪等鍵，重複消費同一事件不會產生重複資料。

## 三、請假簽核流程 (Leave Approval Flow)

此流程展示員工發起請假到主管核准的完整路徑，涉及【考勤模組 (ATT)】與【簽核模組 (WFL)】的協作。

```plantuml
@startuml Leave Approval Flow
|Employee 員工|
start
:開啟「請假申請」頁面;
:選擇假別、起迄日期、事由;
:送出請假申請;

|HR03 Attendance Service|
:接收請假請求;

if (假別餘額是否足夠？) then (不足)
  :回傳錯誤訊息
  「餘額不足，無法申請」;
  |Employee 員工|
  :收到拒絕通知;
  stop
else (足夠)
  :預扣假期餘額（暫扣）;
  :建立 LeaveRequest Aggregate
  （狀態: PENDING）;
  :儲存至 DB;
endif

:發布 **LeaveRequestCreatedEvent**
（request_id, employee_id,
 approver_id, leave_type, dates）;

|HR11 Workflow Service|
:訂閱 LeaveRequestCreatedEvent;
:建立簽核流程實例;
:根據組織結構決定簽核路線
（直屬主管 → 部門主管）;
:推送通知給第一層簽核人;

|Manager 主管|
:收到簽核待辦通知;
:檢視請假明細;

if (核准？) then (駁回)
  :填寫駁回原因;
  |HR11 Workflow Service|
  :更新流程狀態: REJECTED;
  :發布 **LeaveRequestRejectedEvent**;

  |HR03 Attendance Service|
  :訂閱 RejectedEvent;
  :退還預扣假期餘額;
  :更新 LeaveRequest 狀態: REJECTED;

  |Employee 員工|
  :收到駁回通知;
  stop
else (核准)
  :點擊「核准」;
  |HR11 Workflow Service|
  :更新流程狀態: APPROVED;
  :發布 **LeaveRequestApprovedEvent**;
endif

|HR03 Attendance Service|
:訂閱 ApprovedEvent;
:確認扣除假期餘額（正式扣除）;
:更新 LeaveRequest 狀態: APPROVED;
:更新考勤日曆（標記請假日）;

|Employee 員工|
:收到核准通知;
:考勤日曆更新顯示請假;
stop

note right
  **預扣機制：**
  防止員工同時申請
  超額假期（樂觀鎖）

  **簽核路由：**
  由 Workflow 依組織樹
  動態決定簽核鏈
end note
@enduml
```

### 流程設計要點

1. **預扣機制（Reservation Pattern）**：申請時先預扣餘額避免超額申請，駁回時退還。此模式類似電商的庫存預留。
2. **動態簽核路由（Dynamic Routing）**：Workflow 服務根據組織結構樹自動決定簽核鏈，支援多層簽核與代理人機制。
3. **狀態機驅動（State Machine）**：LeaveRequest 的生命週期由狀態機管理：`PENDING → APPROVED/REJECTED`，確保狀態轉換的合法性。

## 四、績效考核流程 (Performance Review Flow)

此流程展示一個完整的績效週期，從 HR 建立考核週期到最終確認評等，涉及【績效模組 (PER)】的多角色協作。

```plantuml
@startuml Performance Review Flow
|HR 人員|
start
:建立考核週期
（名稱、期間、評分權重）;
:選擇參與員工範圍;
:發布考核週期;

|HR08 Performance Service|
:建立 ReviewCycle Aggregate
（狀態: ACTIVE）;
:批次建立 ReviewForm
（每位員工一份表單）;
:發布 **ReviewCycleStartedEvent**;

|Employee 員工|
:收到「自評通知」;
:開啟自評表單;
:填寫各項目自評分數與說明;
:送出自評;

|HR08 Performance Service|
:更新 ReviewForm
（自評分數、狀態: SELF_REVIEWED）;
:通知對應主管;

|Manager 主管|
:收到「待評核通知」;
:檢視員工自評內容;
:填寫主管評分與建議;
:送出主管評核;

|HR08 Performance Service|
:更新 ReviewForm
（主管評分、狀態: MANAGER_REVIEWED）;
:計算加權總分
（自評×30% + 主管×70%）;
:通知 HR 確認;

|HR 人員|
:進入「績效總覽」頁面;
:檢視所有員工評核結果;

if (需要調整？) then (需調整)
  :修改個別評等;
  :填寫調整原因;
else (確認)
endif

:批次確認最終評等;

|HR08 Performance Service|
:鎖定 ReviewCycle
（狀態: FINALIZED）;
:計算各評等分佈
（A/B/C/D 人數比例）;
:發布 **ReviewCycleFinalizedEvent**;

|Employee 員工|
:收到「考核結果通知」;
:查看最終評等與主管建議;
stop

note right
  **評分權重：**
  可由 HR 自訂
  （預設: 自評30% + 主管70%）

  **強制分配：**
  可選擇啟用
  各等級人數比例限制

  **狀態流轉：**
  ACTIVE → SELF_REVIEWED
  → MANAGER_REVIEWED
  → FINALIZED
end note
@enduml
```

### 流程設計要點

1. **彈性表單設計（Flexible Form）**：考核項目與權重由 HR 於週期建立時自定義，支援 KPI、OKR、360 度等多種考核類型。
2. **狀態機嚴格控制（State Machine）**：表單狀態依序流轉 `ACTIVE → SELF_REVIEWED → MANAGER_REVIEWED → FINALIZED`，不可跳步或回退，確保流程正確性。
3. **強制分配機制（Forced Distribution）**：HR 可選擇啟用各評等的人數比例限制（如 A 等不超過 20%），系統會在確認階段提供分佈警示。
