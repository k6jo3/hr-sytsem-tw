# 核心業務流程圖 (Business Flowcharts & Swimlane)

本文件展示系統中跨模組、高複雜度的核心業務邏輯流程。透過泳道圖 (Swimlane Diagram) 清晰界定出不同角色 (Actors) 以及後端微服務集群 (Microservices) 於各個階段應負擔的責任與系統邊界條件。

---

## 一、 考勤結算至薪資發放流程 (Attendance to Payroll Flow)

此流程涵蓋了企業人資系統中最關鍵的「月結處理」。它不僅牽涉前端使用者的互動，更涉及【考勤模組 (ATT)】與【薪資模組 (PAY)】之間的跨服務資料同步與非同步事件運算。

> **圖表格式：** PlantUML 渲染｜原始碼：`04_核心業務流程圖.puml`｜渲染指令：`java -jar tools/plantuml.jar -smetana -o diagrams 04_核心業務流程圖.puml`

![考勤結算至薪資發放流程](diagrams/04_考勤結算至薪資發放流程.png)

### 流程設計要點 (Architecture Design Points)

此設計展示了複雜的分散式結算邏輯如何被穩健地處理：

1. **防呆與依賴檢查 (Constraint & Validation)**：在進入薪資計算（Phase 3）前，必須確保考勤資料已經完全鎖定（Phase 2）。系統防呆設計會阻擋在仍有請假單卡在主管端的狀態下進行結算，確保資料一致性 (Data Integrity)。
2. **事件與 API Query 的混合應用 (Hybrid Communication)**：
   * 考勤結班後會發出 Event 告知 PAY 模組「這段期間的資料已鎖定」。
   * 但計薪時，基於資料即時性且為了避免把龐大運算參數塞入 Kafka 訊息中，PAY 模組是主動呼叫 ATT 與 INS 的 Query API 索取精確的時數與級距數字（API Composition 模式）。此做法降低了 Queue 的負載並提高了資料獲取的正確性。

## 二、員工入職流程 (Employee Onboarding Flow)

此流程描述從 HR 建立員工資料到完成所有系統初始化的跨服務協作。涉及【組織模組 (ORG)】、【IAM 模組】、【保險模組 (INS)】、【薪資模組 (PAY)】四個服務的非同步事件串接。

```plantuml
@startuml Employee Onboarding Flow
|HR 人員|
start
:開啟「新增員工」頁面;
:填寫基本資料
（姓名、身分證、部門、職稱、到職日）;
:送出新增請求;

|HR02 Organization Service|
:驗證資料完整性
（身分證格式、部門存在性）;
:建立 Employee Aggregate;
:儲存至 DB;
:發布 **EmployeeCreatedEvent**
（employee_id, name, department_id, hire_date）;

fork
  |HR01 IAM Service|
  :訂閱 EmployeeCreatedEvent;
  :自動建立使用者帳號
  （username = 員工編號, 預設密碼）;
  :指派 EMPLOYEE 角色;
  :發布 **UserAccountCreatedEvent**;
fork again
  |HR05 Insurance Service|
  :訂閱 EmployeeCreatedEvent;
  :建立勞保投保記錄
  （依投保級距自動計算）;
  :建立健保投保記錄;
  :發布 **InsuranceEnrolledEvent**;
fork again
  |HR04 Payroll Service|
  :訂閱 EmployeeCreatedEvent;
  :建立薪資結構
  （底薪、津貼、扣項）;
  :關聯薪資項目至員工;
  :發布 **SalaryStructureCreatedEvent**;
end fork

|HR 人員|
:收到系統通知「員工入職流程完成」;
:確認各模組資料正確;
stop

note right
  **跨服務通訊模式：**
  Event-Driven（Kafka）

  **最終一致性保證：**
  每個訂閱者獨立處理，
  失敗時進入 DLQ 重試
end note
@enduml
```

### 流程設計要點

1. **事件驅動解耦（Event-Driven Decoupling）**：HR02 只負責建立員工本身，其餘服務透過訂閱 `EmployeeCreatedEvent` 自動完成後續作業，實現服務間零耦合。
2. **最終一致性（Eventual Consistency）**：三個下游服務（IAM、Insurance、Payroll）平行處理，任一服務失敗不影響其他服務，透過 Dead Letter Queue 機制確保最終一致。
3. **冪等性設計（Idempotency）**：每個訂閱者以 `employee_id` 作為冪等鍵，重複消費同一事件不會產生重複資料。

## 三、請假簽核流程 (Leave Approval Flow)

此流程展示員工發起請假到主管核准的完整路徑，涉及【考勤模組 (ATT)】與【簽核模組 (WFL)】的協作。

```plantuml
@startuml Leave Approval Flow
|Employee 員工|
start
:開啟「請假申請」頁面;
:選擇假別、起迄日期、事由;
:送出請假申請;

|HR03 Attendance Service|
:接收請假請求;

if (假別餘額是否足夠？) then (不足)
  :回傳錯誤訊息
  「餘額不足，無法申請」;
  |Employee 員工|
  :收到拒絕通知;
  stop
else (足夠)
  :預扣假期餘額（暫扣）;
  :建立 LeaveRequest Aggregate
  （狀態: PENDING）;
  :儲存至 DB;
endif

:發布 **LeaveRequestCreatedEvent**
（request_id, employee_id,
 approver_id, leave_type, dates）;

|HR11 Workflow Service|
:訂閱 LeaveRequestCreatedEvent;
:建立簽核流程實例;
:根據組織結構決定簽核路線
（直屬主管 → 部門主管）;
:推送通知給第一層簽核人;

|Manager 主管|
:收到簽核待辦通知;
:檢視請假明細;

if (核准？) then (駁回)
  :填寫駁回原因;
  |HR11 Workflow Service|
  :更新流程狀態: REJECTED;
  :發布 **LeaveRequestRejectedEvent**;

  |HR03 Attendance Service|
  :訂閱 RejectedEvent;
  :退還預扣假期餘額;
  :更新 LeaveRequest 狀態: REJECTED;

  |Employee 員工|
  :收到駁回通知;
  stop
else (核准)
  :點擊「核准」;
  |HR11 Workflow Service|
  :更新流程狀態: APPROVED;
  :發布 **LeaveRequestApprovedEvent**;
endif

|HR03 Attendance Service|
:訂閱 ApprovedEvent;
:確認扣除假期餘額（正式扣除）;
:更新 LeaveRequest 狀態: APPROVED;
:更新考勤日曆（標記請假日）;

|Employee 員工|
:收到核准通知;
:考勤日曆更新顯示請假;
stop

note right
  **預扣機制：**
  防止員工同時申請
  超額假期（樂觀鎖）

  **簽核路由：**
  由 Workflow 依組織樹
  動態決定簽核鏈
end note
@enduml
```

### 流程設計要點

1. **預扣機制（Reservation Pattern）**：申請時先預扣餘額避免超額申請，駁回時退還。此模式類似電商的庫存預留。
2. **動態簽核路由（Dynamic Routing）**：Workflow 服務根據組織結構樹自動決定簽核鏈，支援多層簽核與代理人機制。
3. **狀態機驅動（State Machine）**：LeaveRequest 的生命週期由狀態機管理：`PENDING → APPROVED/REJECTED`，確保狀態轉換的合法性。

## 四、績效考核流程 (Performance Review Flow)

此流程展示一個完整的績效週期，從 HR 建立考核週期到最終確認評等，涉及【績效模組 (PER)】的多角色協作。

```plantuml
@startuml Performance Review Flow
|HR 人員|
start
:建立考核週期
（名稱、期間、評分權重）;
:選擇參與員工範圍;
:發布考核週期;

|HR08 Performance Service|
:建立 ReviewCycle Aggregate
（狀態: ACTIVE）;
:批次建立 ReviewForm
（每位員工一份表單）;
:發布 **ReviewCycleStartedEvent**;

|Employee 員工|
:收到「自評通知」;
:開啟自評表單;
:填寫各項目自評分數與說明;
:送出自評;

|HR08 Performance Service|
:更新 ReviewForm
（自評分數、狀態: SELF_REVIEWED）;
:通知對應主管;

|Manager 主管|
:收到「待評核通知」;
:檢視員工自評內容;
:填寫主管評分與建議;
:送出主管評核;

|HR08 Performance Service|
:更新 ReviewForm
（主管評分、狀態: MANAGER_REVIEWED）;
:計算加權總分
（自評×30% + 主管×70%）;
:通知 HR 確認;

|HR 人員|
:進入「績效總覽」頁面;
:檢視所有員工評核結果;

if (需要調整？) then (需調整)
  :修改個別評等;
  :填寫調整原因;
else (確認)
endif

:批次確認最終評等;

|HR08 Performance Service|
:鎖定 ReviewCycle
（狀態: FINALIZED）;
:計算各評等分佈
（A/B/C/D 人數比例）;
:發布 **ReviewCycleFinalizedEvent**;

|Employee 員工|
:收到「考核結果通知」;
:查看最終評等與主管建議;

  **評分權重：**
  可由 HR 自訂
  （預設: 自評30% + 主管70%）

  ACTIVE → SELF_REVIEWED

end note

```

### 流程設計要點

1. **彈性表單設計（Flexible Form）**：考核項目與權重由 HR 於週期建立時自定義，支援 KPI、OKR、360 度等多種考核類型。
2. **狀態機嚴格控制（State Machine）**：表單狀態依序流轉 `ACTIVE → SELF_REVIEWED → MANAGER_REVIEWED → FINALIZED`，不可跳步或回退，確保流程正確性。
3. **強制分配機制（Forced Distribution）**：HR 可選擇啟用各評等的人數比例限制（如 A 等不超過 20%），系統會在確認階段提供分佈警示。

## 五、曠職判定流程 (Absent Detection Flow)

此流程描述系統如何透過每日排程自動偵測未出勤員工並判定曠職，涉及【考勤模組 (ATT)】的排程任務與領域服務協作，以及透過 Domain Event 通知相關服務。

```mermaid
flowchart TD
    Start([每日 19:00 排程觸發<br/>AbsentDetectionJob]) --> A[查詢所有在職員工 ID<br/>employee_read_models]
    A --> A1{取得員工清單<br/>是否成功？}
    A1 -- 失敗 --> A2[降級：從 leave_balances<br/>推導員工清單]
    A2 --> A3{推導是否成功？}
    A3 -- 失敗 --> LogWarn[記錄警告日誌<br/>跳過本日偵測]
    LogWarn --> End1([流程結束])
    A3 -- 成功 --> B
    A1 -- 成功 --> B[查詢今日已有打卡記錄<br/>的員工 ID]

    B --> C[查詢今日已核准請假<br/>涵蓋該日的員工 ID]
    C --> D[AbsentDetectionDomainService<br/>執行缺勤判定]

    D --> D1{員工是否有<br/>打卡記錄？}
    D1 -- 有 --> D5[排除：非缺勤]
    D1 -- 無 --> D2{員工是否有<br/>核准請假？}
    D2 -- 有 --> D5
    D2 -- 無 --> D3[判定為曠職 ABSENT]

    D3 --> E[建立 AttendanceRecord<br/>狀態: ABSENT]
    E --> F[儲存缺勤記錄至 DB<br/>attendanceRecordRepository.save]
    F --> F1{儲存是否成功？}
    F1 -- 失敗 --> F2[記錄錯誤日誌<br/>failCount++<br/>繼續處理下一位]
    F2 --> H
    F1 -- 成功 --> G[發布 AttendanceAnomalyDetectedEvent<br/>anomalyType=ABSENT, anomalyMinutes=0]

    G --> G1[successCount++]
    G1 --> H{還有下一位<br/>缺勤員工？}
    H -- 有 --> E
    H -- 無 --> I[記錄偵測結果統計日誌<br/>在職/已打卡/請假/缺勤/成功/失敗]
    I --> End2([流程結束])

    D5 --> H

    style Start fill:#4CAF50,color:#fff
    style End1 fill:#9E9E9E,color:#fff
    style End2 fill:#4CAF50,color:#fff
    style D3 fill:#F44336,color:#fff
    style G fill:#FF9800,color:#fff
    style F2 fill:#FF5722,color:#fff
    style LogWarn fill:#FF9800,color:#fff
```

### 流程設計要點

1. **排程驅動（Scheduled Job）**：透過 `AbsentDetectionJob` 於每日 19:00 自動執行，以 Spring `@Scheduled(cron = "0 0 19 * * ?")` 宣告，確保每日工作結束後統一偵測。
2. **三層過濾邏輯（Domain Service）**：`AbsentDetectionDomainService` 負責核心判定 —— 從全部在職員工中排除「已打卡」與「已核准請假」者，剩餘即為曠職。此邏輯為純 POJO，不依賴框架。
3. **降級容錯（Fallback Strategy）**：若 `employee_read_models` 表不存在（非 CQRS 環境），自動降級從 `leave_balances` 推導在職員工清單，確保排程不因跨服務表缺失而中斷。
4. **逐筆異常隔離（Per-Record Error Isolation）**：單一員工的缺勤記錄建立失敗不影響其他員工的處理，透過 try-catch 逐筆隔離，最後統計成功/失敗數量。
5. **事件通知（Event-Driven）**：每筆曠職記錄建立後發布 `AttendanceAnomalyDetectedEvent`，下游服務（如通知模組）可訂閱此事件推送給主管。

## 六、薪資預借審核流程 (Salary Advance Approval Flow)

此流程描述員工申請薪資預借，經過資格與額度檢查、主管審核、財務撥款，到每月自動扣回的完整生命週期。涉及【薪資模組 (PAY)】的 SalaryAdvance 聚合根狀態機。

```mermaid
flowchart TD
    Start([員工發起薪資預借申請]) --> A[填寫申請資料<br/>金額、分期月數、事由]
    A --> B[HR04 Payroll Service<br/>接收預借申請]

    B --> V1{驗證：<br/>員工 ID 是否合法？}
    V1 -- 不合法 --> Err1[回傳錯誤：員工 ID 不可為空]
    Err1 --> End1([流程結束])
    V1 -- 合法 --> V2{驗證：<br/>申請金額 > 0？}
    V2 -- 否 --> Err2[回傳錯誤：申請金額必須 > 0]
    Err2 --> End1
    V2 -- 是 --> V3{驗證：<br/>分期月數 >= 1？}
    V3 -- 否 --> Err3[回傳錯誤：分期月數必須 >= 1]
    Err3 --> End1
    V3 -- 是 --> V4[計算可預借上限<br/>公式: 應發薪資 - 法定扣除 - 法扣款 × 90%]

    V4 --> V5{申請金額 <=<br/>可預借上限？}
    V5 -- 超過 --> Err4[回傳錯誤：超過可預借上限]
    Err4 --> End1
    V5 -- 通過 --> C[建立 SalaryAdvance Aggregate<br/>狀態: PENDING]
    C --> D[儲存至 DB<br/>通知主管審核]

    D --> E[主管收到審核通知]
    E --> F{主管審核決定}
    F -- 駁回 --> G[主管填寫駁回原因]
    G --> G1[SalaryAdvance.reject<br/>狀態: PENDING → REJECTED]
    G1 --> G2[通知員工：預借申請被駁回]
    G2 --> End2([流程結束])

    F -- 核准 --> H[主管核定核准金額<br/>可調整為低於申請金額]
    H --> H1{核准金額 > 0<br/>且 <= 申請金額？}
    H1 -- 否 --> Err5[回傳錯誤：核准金額不合法]
    Err5 --> End1
    H1 -- 是 --> H2[SalaryAdvance.approve<br/>狀態: PENDING → APPROVED]
    H2 --> H3[計算每期扣回金額<br/>= 核准金額 / 分期月數 向上取整]
    H3 --> I[通知財務部門]

    I --> J[財務確認撥款]
    J --> J1[SalaryAdvance.disburse<br/>狀態: APPROVED → DISBURSED<br/>記錄撥款日]
    J1 --> K[撥款至員工帳戶]
    K --> L[通知員工：預借已撥款]

    L --> M([每月薪資計算觸發扣回])
    M --> N[SalaryAdvance.repay<br/>扣除 installmentAmount]
    N --> N1{remainingBalance<br/>= 0？}
    N1 -- 否 --> N2[狀態: REPAYING<br/>下月繼續扣回]
    N2 --> M
    N1 -- 是 --> O[狀態: REPAYING → FULLY_REPAID<br/>預借完全清償]
    O --> End3([流程結束])

    style Start fill:#2196F3,color:#fff
    style End1 fill:#9E9E9E,color:#fff
    style End2 fill:#F44336,color:#fff
    style End3 fill:#4CAF50,color:#fff
    style C fill:#FF9800,color:#fff
    style H2 fill:#4CAF50,color:#fff
    style J1 fill:#2196F3,color:#fff
    style O fill:#4CAF50,color:#fff
    style G1 fill:#F44336,color:#fff
```

### 流程設計要點

1. **聚合根狀態機（Aggregate State Machine）**：`SalaryAdvance` 聚合根嚴格管控狀態轉換 `PENDING → APPROVED → DISBURSED → REPAYING → FULLY_REPAID`，每個方法在入口處驗證當前狀態，不合法的轉換拋出 `IllegalStateException`。另有 `REJECTED` 與 `CANCELLED` 兩個終態。
2. **預借額度公式（Max Advance Calculation）**：`可預借金額 <= (應發薪資 - 法定扣除 - 法扣款) × 90%`，確保員工扣回預借後仍有基本薪資保障。此公式以靜態方法 `SalaryAdvance.calculateMaxAdvance()` 實現。
3. **分期扣回機制（Installment Repayment）**：核准後自動計算 `installmentAmount = approvedAmount / installmentMonths`（向上取整），每月薪資計算時自動呼叫 `repay()` 扣回，最後一期扣回金額 = `remainingBalance`，確保不多扣。
4. **核准金額可調整（Adjustable Approval）**：主管可核准低於申請金額的額度（但不可超過申請金額），提供彈性審核空間。
5. **取消限制（Cancel Constraint）**：僅 `PENDING` 與 `APPROVED` 狀態可取消，已撥款（`DISBURSED`）或扣回中（`REPAYING`）的預借不可取消，保護財務一致性。

## 七、離職退保連動流程 (Termination Insurance Withdrawal Flow)

此流程描述員工離職生效後，系統自動觸發退保的跨服務事件驅動流程。涉及【組織模組 (ORG)】、【保險模組 (INS)】、【IAM 模組】、【薪資模組 (PAY)】四個服務的非同步協作。

```mermaid
flowchart TD
    Start([HR 執行員工離職作業]) --> A[HR02 Organization Service<br/>TerminateEmployeeServiceImpl<br/>啟動 Pipeline]

    A --> P1[Step 1: LoadEmployeeTask<br/>載入員工資料]
    P1 --> P2[Step 2: TerminateEmployeeTask<br/>呼叫 Employee.terminate<br/>設定離職日期與原因]
```

### 流程設計要點

1. **Pipeline 編排（Business Pipeline）**：離職作業由 `TerminateEmployeeServiceImpl` 透過 `BusinessPipeline` 依序執行 5 個 Task（載入 → 離職 → 儲存 → 歷程 → 事件），符合 Application Service 只做編排不做決策的原則。
2. **事件驅動跨服務連動（Event-Driven Cross-Service）**：`EmployeeTerminatedEvent` 攜帶 `employeeId`、`terminationDate`、`terminationReason` 等完整資訊，透過 Kafka topic `hrms.employee.terminated` 廣播，保險/IAM/薪資服務各自訂閱處理。
3. **退保日期規則引擎（EffectiveDateRule）**：依保險類型自動計算退保生效日 —— 勞保/勞退/團保為離職日退保；健保特殊規則：月底離職者次月 1 日退保。此規則封裝於 `EffectiveDateRule.calculateWithdrawDate()` 值物件中。
4. **逐筆隔離與降級（Per-Record Isolation）**：每筆加保記錄的退保獨立處理，單筆退保失敗不影響其他記錄，透過 try-catch 隔離錯誤並記錄日誌。
5. **雙層事件鏈（Two-Layer Event Chain）**：第一層 `EmployeeTerminatedEvent`（ORG → INS）觸發退保；第二層 `InsuranceWithdrawalCompletedEvent`（INS → PAY）通知薪資模組移除保費扣款項目，實現完整的事件驅動解耦。
6. **冪等性保障（Idempotency）**：`AutoWithdrawOnTerminationService` 查詢的是 `ACTIVE` 狀態的加保記錄，已退保（`WITHDRAWN`）的記錄不會被重複處理，確保事件重複消費的安全性。
