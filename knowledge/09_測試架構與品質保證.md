# 測試架構與品質保證 (Testing Architecture & Quality Assurance)

本文件說明 HRMS 系統的測試策略、自研合約驅動測試框架的設計原理，以及各服務的測試覆蓋成果。

---

## 一、 測試金字塔策略 (Testing Pyramid)

```
                  ┌─────────────────┐
                  │  整合測試 (E2E)  │  ← API 端點層級，驗證完整流程
                  │    少量 + 慢     │
                ┌─┴─────────────────┴─┐
                │   合約驅動測試       │  ← ★ 本系統核心，SA 規格即測試
                │  (Contract Tests)   │
              ┌─┴─────────────────────┴─┐
              │   Domain 單元測試        │  ← 業務規則、Value Object 驗證
              │   (Unit Tests)          │
            ┌─┴─────────────────────────┴─┐
            │   Frontend 工廠/Hook 測試    │  ← DTO 轉換、React Hook 行為
            │   (Vitest + RTL)            │
            └─────────────────────────────┘
```

本系統的測試重點放在**中間兩層**：合約驅動測試與 Domain 單元測試，而非耗時的端對端測試。核心理念是：**越靠近業務規格的斷言，越有價值。**

## 二、 合約驅動測試框架（自研）

### 2.1 設計動機

傳統整合測試由工程師自行定義，驗證的是「工程師對需求的理解」而非「SA 定義的業務規格」。兩者之間的落差是品質問題的根源。

| 面向 | 傳統整合測試 | 合約驅動測試 |
|:---|:---|:---|
| 誰定義測試標準 | 工程師（自己測試自己）| SA（外部規格驗證）|
| 需求文件與測試的關係 | 脫節，各自獨立維護 | **需求文件即測試依據** |
| 發現的問題類型 | 程式邏輯錯誤 | 業務意圖不符 |
| 維護成本 | 需求變更時大幅改寫 | 只需更新 Markdown 表格 |

### 2.2 框架架構

```
contracts/
├── iam_contracts.md          ← SA 定義：場景ID、過濾條件、回應規格
├── organization_contracts.md
├── attendance_contracts.md
└── ... (14 服務合約)

backend/hrms-common/
├── MarkdownContractEngine.java  ← 解析 Markdown 表格，提取合約規格
└── BaseContractTest.java        ← 提供 assertContract() 斷言方法

backend/hrms-{service}/
└── *ApiContractTest.java        ← 各服務的合約測試實作
```

### 2.3 合約規格格式（SA 撰寫）

SA 以標準 Markdown 表格定義業務場景，每一行對應一個測試案例：

```markdown
| 場景 ID   | 測試描述         | 模擬角色 | API 端點                   | 必須包含的過濾條件                                    |
| :---      | :---             | :---     | :---                       | :---                                                 |
| EMP_001   | 一般員工查詢在職 | EMPLOYEE | GET /api/v1/employees      | `status = 'ACTIVE'`, `is_deleted = 0`               |
| EMP_002   | HR 查詢特定部門  | HR       | GET /api/v1/employees      | `department_id = 'D001'`, `is_deleted = 0`           |
| ATT_001   | 查詢本月打卡紀錄 | EMPLOYEE | GET /api/v1/attendance     | `employee_id = {currentUserId}`, `month = '2026-02'` |
```

### 2.4 三層驗證能力

`BaseContractTest` 的 `assertContract()` 方法驗證三個層次：

```
Layer 1: 查詢條件驗證（Query Filters）
├── 所有必要的過濾條件是否存在？
├── 運算子是否正確（=、LIKE、>=、IN、IS NULL）？
└── 值是否符合預期（含動態佔位符替換）？

Layer 2: 回應結構驗證（Response Structure）
├── HTTP 狀態碼是否正確？
├── 必要欄位是否存在（requiredFields）？
└── 分頁資訊是否完整（pagination）？

Layer 3: 副作用驗證（Side Effects，Command 測試）
├── 資料庫是否有正確的 INSERT/UPDATE 操作？
└── Domain Event 是否正確發布（eventType + payload）？
```

### 2.5 測試程式碼範例

```java
// 合約測試只需三步驟：載入合約 → 執行操作 → 斷言
@Test
@DisplayName("EMP_001: 一般員工查詢在職人員")
void searchEmployee_AsEmployee_ShouldIncludeSecurityFilters() {
    // 1. 載入 SA 定義的合約規格
    String contractSpec = loadContractSpec("organization");

    // 2. 組裝查詢條件
    var request = EmployeeSearchRequest.builder()
        .status("ACTIVE")
        .build();

    // 3. 攔截 QueryGroup 並斷言（不執行真實 SQL）
    var queryGroup = assembler.toQueryGroup(request, mockCurrentUser());
    assertContract(queryGroup, contractSpec, "EMP_001");
    // ↑ 內部驗證：status=ACTIVE ✓、is_deleted=0 ✓、tenant_id=T001 ✓
}
```

**Command 測試範例（驗證資料異動 + 領域事件）：**

```java
@Test
@DisplayName("TRN_CMD_001: 建立教育訓練課程")
void createCourse_ShouldSaveCourseAndPublishEvent() {
    // 準備 & 執行
    performPost("/api/v1/training/courses", createRequest)
        .andExpect(status().isOk());

    // 驗證三層：
    // Layer 1: Repository 是否被正確呼叫
    verify(courseRepository).save(argThat(course ->
        "Java 進階開發".equals(course.getTitle()) &&
        CourseStatus.DRAFT == course.getStatus()
    ));

    // Layer 3: Domain Event 是否發布
    verify(eventPublisher).publish(argThat(event ->
        event instanceof CourseCreatedEvent
    ));
}
```

## 三、 前端測試架構（TDD）

前端採用 **Test-Driven Development**，以 Vitest + React Testing Library 實作。

### 3.1 Factory 測試（強制要求）

每個 `ViewModelFactory` 都有對應的單元測試，驗證 DTO → ViewModel 的轉換邏輯：

```typescript
describe('EmployeeViewModelFactory', () => {
  it('should transform status to Chinese label', () => {
    const dto = { id: '1', status: 'ACTIVE', first_name: 'John', last_name: 'Doe' };
    const vm = EmployeeViewModelFactory.createFromDTO(dto);

    expect(vm.fullName).toBe('John Doe');
    expect(vm.statusLabel).toBe('在職');    // ✅ 不是 'ACTIVE'
    expect(vm.isActive).toBe(true);
  });
});
```

**設計理念**：Factory 層是前後端介面的緩衝區，集中處理所有格式轉換。測試 Factory 等同於測試前後端契約，變更 API 格式只需修改 Factory 與其測試。

### 3.2 Hook 測試

```typescript
describe('useEmployeeDetail', () => {
  it('should return transformed viewModel when API succeeds', async () => {
    server.use(rest.get('/api/v1/employees/1', (req, res, ctx) =>
      res(ctx.json(mockEmployeeDto))
    ));

    const { result } = renderHook(() => useEmployeeDetail('1'));
    await waitFor(() => expect(result.current.loading).toBe(false));

    expect(result.current.employee?.fullName).toBe('John Doe');
    // ↑ 驗證 Hook 使用了 Factory 轉換，而非直接回傳 DTO
  });
});
```

### 3.3 Component 測試

```typescript
describe('AttendanceClockCard', () => {
  it('should show clock-in button when employee has not clocked in today', () => {
    render(<AttendanceClockCard status="NOT_CLOCKED_IN" />);
    expect(screen.getByRole('button', { name: '上班打卡' })).toBeInTheDocument();
    expect(screen.queryByRole('button', { name: '下班打卡' })).not.toBeInTheDocument();
  });
});
```

## 五、 關鍵設計亮點

### 亮點一：需求文件與測試程式碼一體化

```
傳統做法：
  需求文件（Word/Confluence）  →  工程師閱讀  →  自行撰寫測試
  （兩份文件，永遠可能不同步）

本系統做法：
  contracts/*.md（SA 撰寫）  →  MarkdownContractEngine 解析  →  assertContract() 斷言
  （一份文件，需求即是測試的 ground truth）
```

### 亮點二：QueryGroup 可序列化，測試不依賴 SQL

傳統方式測試查詢邏輯，需要啟動資料庫並比對 SQL 字串（脆弱、慢）。

本系統的 `QueryGroup` 是**結構化物件**，Assembler 的輸出可以直接做欄位級別的斷言，不需要 SQL 字串比對：

```java
// ❌ 傳統：比對 SQL 字串（脆弱）
assertThat(capturedSql).contains("WHERE status = 'ACTIVE'");

// ✅ 本系統：比對結構化物件（穩健）
assertThat(queryGroup.getFilters())
    .anyMatch(f -> "status".equals(f.getField())
               && "ACTIVE".equals(f.getValue())
               && Operator.EQ == f.getOperator());
```

### 亮點三：合約測試揭露 SA 規格與 Domain 不一致

合約測試在 HR07 開發時發現：SA 合約定義 `status = 'PENDING'`，但 Domain 實際使用 `SUBMITTED`。

這個不一致在傳統測試中可能永遠不會被發現（因為工程師測試自己寫的 Domain）。合約驅動測試讓 SA 規格成為**獨立的驗收標準**，提前暴露業務語義分歧。

### 亮點四：Frontend Factory 作為前後端的防火牆

```
後端 API 回傳格式變更
        ↓
  只需修改 Factory（1 個檔案）
        ↓
  所有依賴此 Factory 的 UI 元件自動相容
        ↓
  Factory 測試失敗 → 立即定位問題
```

## 六、 如何執行測試

### 後端合約測試

```bash
# 單一服務的合約測試
cd backend/hrms-{service}
mvn test -Dtest=*ContractTest

# 所有服務一次執行（從根目錄）
cd backend
mvn test -Dtest=*ContractTest --fail-at-end

# 執行前確認 hrms-common 已安裝
cd backend/hrms-common
mvn clean install -DskipTests
```

### 前端測試

```bash
cd frontend

# 執行全部測試
npm run test

# 監看模式（開發時使用）
npm run test:watch

# 顯示覆蓋率報告
npm run test:coverage
```
