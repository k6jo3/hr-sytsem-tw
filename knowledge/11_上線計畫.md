### 0.2 docker-compose.yml 骨架

```yaml

services:
  postgres:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: hrms_app
      POSTGRES_PASSWORD: local_dev_password
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./scripts/init-databases.sql:/docker-entrypoint-initdb.d/init.sql

  hrms-iam:
    build:
      context: ./backend
      dockerfile: hrms-iam/Dockerfile
    environment:
      SPRING_PROFILES_ACTIVE: local
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/hrms_iam
      SPRING_DATASOURCE_PASSWORD: local_dev_password
      APP_JWT_SECRET: local_dev_jwt_secret_must_be_at_least_64_chars_long_for_security
    ports:
      - "8081:8080"
    depends_on:
      postgres:
        condition: service_healthy

  # 依此格式新增 hrms-organization（port 8082）、hrms-attendance（port 8083）

volumes:
  postgres_data:
```

### 1.2 服務部署順序（重要！）

```
IAM 服務（先部署）
    ↓ 其他服務需要 IAM 驗證
ORG 服務
    ↓ 考勤需要員工資料
ATT 服務
```

**部署各服務的指令模板：**

```bash
# 共用變數
PROJECT_ID="hrms-portfolio"
REGION="asia-east1"
REPO="asia-east1-docker.pkg.dev/${PROJECT_ID}/hrms-repo"
DB_PRIVATE_IP="10.xx.xx.xx"  # Phase 1.1 Step 1.5 查到的 IP

# 1. 建置並推送 IAM Image
cd D:/git/hr-system2/hr-sytsem-2/backend
IMAGE="${REPO}/hrms-iam"
docker build -f hrms-iam/Dockerfile -t ${IMAGE}:latest .
docker push ${IMAGE}:latest

# 2. 部署 IAM 到 Cloud Run
gcloud run deploy hrms-iam-service \
  --image=${IMAGE}:latest \
  --region=${REGION} \
  --allow-unauthenticated \
  --min-instances=0 \
  --max-instances=3 \
  --memory=512Mi \
  --vpc-connector=hrms-connector \
  --set-env-vars="SPRING_PROFILES_ACTIVE=prod,SPRING_DATASOURCE_URL=jdbc:postgresql://${DB_PRIVATE_IP}:5432/hrms_iam,SPRING_DATASOURCE_USERNAME=hrms_app" \
  --set-secrets="SPRING_DATASOURCE_PASSWORD=DB_PASSWORD:latest,APP_JWT_SECRET=JWT_SECRET:latest"

# 3. 取得 IAM URL（後續 ORG/ATT 需要填入）
IAM_URL=$(gcloud run services describe hrms-iam-service \
  --region=${REGION} --format="value(status.url)")
echo "IAM URL: ${IAM_URL}"

# 4. 依樣部署 ORG（換 hrms-organization、hrms_org DB）
# 5. 依樣部署 ATT（換 hrms-attendance、hrms_att DB）
```

### 2.2 Kafka on GCP

> **選項 A：Confluent Cloud（推薦作品集使用）**
> - 免費 tier：$0（前 30 天），後續約 $15/月
> - 無需自建 Kafka cluster，設定最簡單
> - 適合 Portfolio 展示用

> **選項 B：Google Pub/Sub（替代方案）**
> - 費用更低，原生 GCP 整合
> - 需要修改程式碼（Kafka API → Pub/Sub SDK）

```bash
# Confluent Cloud 設定方式（選項 A）
# 1. 前往 https://confluent.cloud 免費註冊
# 2. 建立 Cluster（選 Basic / Standard）
# 3. 取得 Bootstrap Server + API Key + API Secret
# 4. 存入 Secret Manager
echo -n "pkc-xxxxx.asia-east1.gcp.confluent.cloud:9092" | \
  gcloud secrets create KAFKA_BOOTSTRAP_SERVERS --data-file=-

# 5. 部署時注入 Kafka 設定
gcloud run deploy hrms-timesheet-service \
  --set-secrets="KAFKA_BOOTSTRAP_SERVERS=KAFKA_BOOTSTRAP_SERVERS:latest" \
  --set-secrets="KAFKA_API_KEY=KAFKA_API_KEY:latest" \
  --set-secrets="KAFKA_API_SECRET=KAFKA_API_SECRET:latest"
```

### 3.1 GitHub Actions CI/CD

**Trigger 策略：**

```
推送到 main branch
    ↓
觸發 GitHub Actions

每個服務獨立 Workflow（只在服務有變更時才執行）：
├── hrms-iam-deploy.yml     ← 偵測 backend/hrms-iam/** 變更
├── hrms-organization-deploy.yml
├── hrms-attendance-deploy.yml
└── ...（共 14 個 workflow 檔案）

或使用 Path Filter + Matrix 方式（進階）：
└── backend-deploy.yml      ← 統一 workflow，matrix 選擇服務
```

**CI/CD 工作流程檔案參考：** `06_CICD與系統部署指南.md`

### 3.3 Cloud Monitoring 設定

```bash
# 1. 啟用 Cloud Monitoring API
gcloud services enable monitoring.googleapis.com

# 2. 建立 Uptime Check（每 5 分鐘 ping 服務）
gcloud monitoring uptime-check-configs create \
  --display-name="IAM Health Check" \
  --http-check-path="/actuator/health" \
  --monitored-resource-type="cloud-run-revision" \
  --period=300

# 3. 建立 Alert Policy（連續 2 次 check 失敗才通知）
# 建議在 GCP Console 用 GUI 設定較直覺
# Monitoring → Alerting → Create Policy
```

### 3.4 Spring Boot Actuator 設定

每個服務的 `application-prod.yml` 需加入：

```yaml

  endpoints:
    web:
      exposure:
        include: health,info,metrics
  endpoint:
    health:
      show-details: when-authorized
```

## 常見卡關點與解決方向

| 卡關情境 | 優先檢查 |
|:---|:---|
| Docker Image 推送失敗 | `gcloud auth configure-docker asia-east1-docker.pkg.dev` |
| Cloud Run 啟動後 crash | `gcloud logging read` 查 Spring Boot 啟動日誌，通常是 DB 連線失敗或 OOM |
| OOM（Out of Memory） | `--memory=1Gi` 解決；Java 21 需要 ≥ 512Mi，建議 1Gi |
| DB 連線被拒 | 確認 VPC Connector 狀態為 READY，且 `--vpc-connector` 已設定 |
| Secret 取不到值 | Cloud Run Service Account 需要 `secretmanager.secretAccessor` 角色 |
| Kafka 連線超時 | 確認 Confluent Cloud Cluster 已啟動（非 Paused 狀態） |
| GitHub Actions 403 | 確認 `GCP_CREDENTIALS` 內容完整，Service Account 有 `run.admin` 角色 |

## 快速參考：部署指令速查

```bash
# 查看所有 Cloud Run 服務狀態
gcloud run services list --region=asia-east1

# 查看特定服務 URL
gcloud run services describe {service-name} \
  --region=asia-east1 --format="value(status.url)"

# 查看服務 Log（最近 50 筆）
gcloud logging read \
  "resource.type=cloud_run_revision AND resource.labels.service_name={service-name}" \
  --limit=50 --format="value(textPayload)"

# 更新服務記憶體
gcloud run services update {service-name} \
  --region=asia-east1 --memory=1Gi

# 強制重新部署（拉最新 Image）
gcloud run deploy {service-name} \
  --image={image-url}:latest \
  --region=asia-east1

# 刪除服務（節省費用）
gcloud run services delete {service-name} --region=asia-east1
```
