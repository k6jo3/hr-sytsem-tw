## 一、2 分鐘版本（電梯簡報）

> **適用：** 開場自我介紹、面試官問「簡單介紹一下你的作品」

---

### 開場（20 秒）

> *「我做了一個針對 500 人規模企業的**人資暨專案管理系統**，用微服務架構把過去企業裡各自獨立的考勤、薪資、招募和專案追蹤整合在一起。」*

---

### 系統規模（20 秒）

> *「整個系統分成 **14 個微服務**，用 DDD 的 Bounded Context 來劃分邊界，後端是 Spring Boot + Java 21，前端是 React + TypeScript，部署在 GCP Cloud Run 上。」*

---

### 技術亮點（40 秒）

> *「我特別想說三個技術選擇：*
> *第一，**服務之間完全透過 Kafka 非同步解耦**，比如員工到職後，IAM 自動建帳號、INS 自動加保，HR 只需要填一張表。*
> *第二，薪資結算用了 **SAGA Pattern** 處理分散式交易，確保跨服務的資料一致性。*
> *第三，我設計了一套**合約驅動測試框架**，讓 SA 用 Markdown 定義業務規格，工程師的測試直接對規格做斷言，把需求文件變成可執行的測試。」*

---

### 結尾（20 秒）

> *「目前前端 14 個功能模組已全部完整實作並通過 TDD 測試，後端合約測試也覆蓋了 14 個服務的主要查詢和命令場景。此外，系統完成了全面的 UI/UX 改版——深色 Sidebar、登入頁重設計、統一 PageHeader，讓整體視覺更具專業感。如果你有興趣，我可以針對某個技術點深入說明。」*

## 二、5 分鐘版本（完整導覽）

> **適用：** 面試官說「詳細介紹一下這個系統」，搭配圖表說明

---

### 第 1 分鐘：系統背景與問題定義

> *「先說這個系統在解決什麼問題。*
>
> *傳統企業的人資通常是多套系統並行——考勤用一套、薪資用另一套、專案追蹤又是另一套，員工資料要重複維護。我設計這個系統的目標，是建立一個 **Single Source of Truth 的人資入口**，讓員工從入職到離職的所有資料流都在同一個平台上流轉。」*

**👉 打開 `diagrams/03_usecase.png`**

> *「這張 Use Case 圖把三種角色的操作範圍分得很清楚：一般員工有 ESS 自助服務；主管有審核和績效考核；HR 負責系統層級的參數維護和薪資結算。這 12 個 Use Case 背後，就是 14 個微服務在支撐。」*

---

### 第 2 分鐘：系統架構

**👉 打開 `diagrams/00_architecture-1.png`**

> *「這張架構圖從上到下有三層：*
>
> *最上層是前端，三種入口對應三種角色。中間是 14 個微服務，我用 DDD 把它們分成四個領域：核心領域（IAM、組織）、HR 領域（考勤、薪資、保險）、專案績效領域、招募人才領域，還有底部的支援服務。*
>
> *這張圖最重要的，是中間這條 **Kafka Event Bus**。服務之間的溝通不是同步 REST 呼叫，而是透過領域事件非同步驅動。比如這裡——ORG 服務 publish `EmployeeCreatedEvent`，下游的 IAM、PAY、NTF 各自訂閱、各自處理，完全解耦。」*

---

### 第 3 分鐘：兩個核心技術決策

**決策一：SAGA Pattern（薪資結算）**

**👉 打開 `diagrams/01_sequence-1.png`（薪資 SAGA 段落）**

> *「薪資結算這個業務最複雜，它需要跨 ATT、INS、PAY 三個服務聚合資料，然後計算、鎖定、產出媒體檔。任何一步失敗都要補償回去。*
>
> *我用 SAGA Pattern 把這個長流程拆成一系列有補償機制的步驟，確保不會出現部分成功的髒狀態。比如薪資媒體檔產出失敗，系統會自動觸發補償交易，把 PAY 的鎖定狀態回滾。」*

**決策二：每個服務獨立資料庫（Database-per-Service）**

> *「服務之間禁止跨庫 JOIN，資料共用一律透過 API 或事件。雖然查詢複雜度增加了，但換來的是每個服務可以獨立演進、獨立部署，不會因為一個 schema 變更影響全系統。」*

---

### 第 4 分鐘：測試策略——合約驅動測試

> *「這是我在這個專案裡最想分享的設計。*
>
> *傳統做法，SA 寫需求文件，工程師自己理解後寫測試，測試內容和需求文件是脫節的。我設計了一套**合約驅動測試框架**：*
>
> *SA 用 Markdown 表格定義每個 API 場景的業務規格——輸入什麼、預期的查詢過濾條件是什麼、回應欄位應該有什麼。這份 Markdown 就是合約。*
>
> *工程師的測試則直接載入這份合約做斷言：驗證 Assembler 是否正確組裝了查詢條件，驗證 Response 結構是否符合規格。需求文件本身變成了可執行的測試，需求和實作不可能再脫節。」*

> *「目前 14 個服務的合約測試累計超過 200 個場景，全部通過。」*

---

### 第 5 分鐘：前端實作與開發規範

> *「前端用 React + TypeScript，採用 Feature-based 模組化結構。*
>
> *我強制規定 API 回應不能直接進 UI，**一律透過 Factory 轉換成 ViewModel**。這樣做有兩個好處：一是前後端介面變更時只需改 Factory，不影響 UI 邏輯；二是 Factory 的單元測試非常好寫，快照測試一行搞定。*
>
> *目前 14 個功能模組全部完整實作，83 個測試檔案、1241 個測試案例全數通過，採用 TDD 開發流程，每個 Factory、Hook、Component 都有對應的測試。」*

> *「此外，我完成了一次全面的 UI/UX 改版。重點包括：**深色 Sidebar** 提升視覺層次、**登入頁重設計** 強化品牌識別、**統一 PageHeader** 確保 9 個主要頁面一致的操作體驗、頁面動畫與通知鈴鐺增強互動感。這次改版展現了在功能開發完成後，持續打磨使用者體驗的能力。」*

> *「整體來說，這個作品的重點不只是功能完整，而是架構設計的嚴謹性——從 DDD 分層、事件驅動、到可執行的合約測試，每一層都有明確的邊界和可驗證的規範。」*

## 三、常見問答準備 (Q&A)

### 架構類

**Q：為什麼選微服務而不是 Monolith？**
> *「這個系統的業務領域天然就是分散的——考勤的部署頻率和薪資完全不同，薪資結算月底才跑一次但需要很強的計算資源，而考勤打卡是全天高頻請求。用微服務可以讓各模組獨立擴展。當然代價就是複雜度，所以才需要 Kafka 和 SAGA 來管理跨服務協作。」*

**Q：14 個服務是怎麼劃分的？劃分標準是什麼？**
> *「用 DDD 的 Bounded Context。劃分時有三個判斷標準：第一，這個領域的核心業務邏輯是否獨立？第二，這個服務的部署週期和擴展需求是否和其他服務不同？第三，這個領域的資料模型是否可以完全自主？這三個都滿足，才值得獨立成一個服務。」*

**Q：服務這麼多，怎麼防止分散式系統的資料不一致？**
> *「兩個機制：一是 Kafka 的 Event Sourcing，每個狀態變更都是一個不可變的事件，消費端可以重播；二是關鍵的跨服務業務流程用 SAGA，每個步驟都有補償交易，確保最終一致性。我們接受短暫的不一致，但不接受永久的不一致。」*

---

### 技術選擇類

**Q：為什麼用 Kafka 而不是 RabbitMQ 或直接 REST？**
> *「三個原因：持久化（事件不會因為消費端宕機而遺失）、重播能力（新服務可以從歷史事件追回狀態）、高吞吐量（考勤打卡尖峰期一次可能幾百人同時操作）。RabbitMQ 適合任務佇列，Kafka 更適合事件流。」*

**Q：CQRS 實際上怎麼在你的系統裡落地的？**
> *「Controller 分 CmdController 和 QryController 兩支。Command 走 CommandApiService，涉及 domain 邏輯後發布事件；Query 走 QueryApiService，用 QueryDSL 的宣告式查詢引擎直接查 Read Model，不經過 Domain 層，速度快也不影響寫入效能。」*

**Q：合約測試框架是自己寫的嗎？和 Pact 有什麼不同？**
> *「是的，自己設計的。Pact 是 Consumer-Driven Contract，側重服務間 API 格式的相容性。我的框架是 Business-Driven Contract，側重『這個 API 是否符合 SA 定義的業務規格』——查詢條件是否正確組裝、回應結構是否包含必要欄位、命令操作是否觸發正確的領域事件。兩者目標不同，我的框架更貼近業務驗證。」*

---

### 挑戰與反思類

**Q：開發過程中遇到最難的問題是什麼？**
> *「薪資計算的 SAGA 補償機制設計。薪資結算分好幾個步驟，考勤資料拉取、保費計算、最終淨薪計算、媒體檔產出，每個步驟失敗的補償邏輯都不同。特別是已經部分計算的中間狀態怎麼定義和回滾，花了最多時間設計。」*

**Q：如果重做一次，你會做什麼不同的選擇？**
> *「可能會提早引入 API Gateway 統一處理認證和限流，目前每個服務都要處理 JWT 驗證有些重複。另外合約測試框架的 Markdown 格式雖然對 SA 友善，但 JSON Schema 格式的表達能力更強，後來也支援了兩種格式並存。」*

**Q：這個系統上線了嗎？有多少用戶？**
> *「這是個人作品集專案，主要目的是展示架構設計和技術選型的能力。系統有完整的設計文件、可執行的測試覆蓋、以及 Docker Compose 本地啟動環境，更接近一個可交付的系統設計而不是玩具專案。」*
