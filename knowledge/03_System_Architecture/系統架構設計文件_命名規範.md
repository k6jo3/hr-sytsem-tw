```java
/**
 * IAM - 使用者管理 Command Controller
 * 負責使用者的新增、修改、刪除等寫入操作
 */
@RestController
@RequestMapping("/api/v1/users")
@Tag(name = "HR01-User-Command", description = "使用者管理寫入操作")
public class HR01UserCmdController extends CommandBaseController {
    
    @Operation(summary = "新增使用者", operationId = "createUser")
    @PostMapping
    public ResponseEntity<CreateUserResponse> createUser(
            @RequestBody @Valid CreateUserRequest request,
            @CurrentUser JWTModel currentUser) throws Exception {
        return ResponseEntity.ok(execCommand(request, currentUser));
    }
    
    @Operation(summary = "更新使用者", operationId = "updateUser")
    @PutMapping("/{userId}")
    public ResponseEntity<UpdateUserResponse> updateUser(
            @PathVariable String userId,
            @RequestBody @Valid UpdateUserRequest request,
            @CurrentUser JWTModel currentUser) throws Exception {
        return ResponseEntity.ok(execCommand(request, currentUser, userId));
    }
    
    @Operation(summary = "停用使用者", operationId = "deactivateUser")
    @PutMapping("/{userId}/deactivate")
    public ResponseEntity<Void> deactivateUser(
            @PathVariable String userId,
            @CurrentUser JWTModel currentUser) throws Exception {
        execCommand(new DeactivateUserRequest(), currentUser, userId);
        return ResponseEntity.noContent().build();
    }
}

/**
 * IAM - 使用者管理 Query Controller
 * 負責使用者的查詢操作
 */
@RestController
@RequestMapping("/api/v1/users")
@Tag(name = "HR01-User-Query", description = "使用者管理查詢操作")
public class HR01UserQryController extends QueryBaseController {
    
    @Operation(summary = "查詢使用者列表", operationId = "getUserList")
    @GetMapping
    public ResponseEntity<Page<UserListResponse>> getUserList(
            @ModelAttribute UserQueryRequest request,
            @CurrentUser JWTModel currentUser) throws Exception {
        return ResponseEntity.ok(getResponse(request, currentUser));
    }
    
    @Operation(summary = "查詢單一使用者", operationId = "getUser")
    @GetMapping("/{userId}")
    public ResponseEntity<UserDetailResponse> getUser(
            @PathVariable String userId,
            @CurrentUser JWTModel currentUser) throws Exception {
        return ResponseEntity.ok(getResponse(new GetUserRequest(), currentUser, userId));
    }
}
```

#### 5.5.3 Query與Command Controller拆分策略分析

**🔍 策略比較:**

| 面向 | 整合式 (單一Controller) | 拆分式 (Query/Command分離) |
|:---|:---|:---|
| **CQRS原則** | ❌ 違反CQRS | ✅ 完全符合CQRS |
| **單一職責** | ❌ 混合讀寫職責 | ✅ 職責清晰分離 |
| **程式碼組織** | 較簡單 | 類別數量增加 |
| **擴展性** | 較難獨立擴展 | 讀寫可獨立擴展 |
| **團隊協作** | 容易衝突 | 可平行開發 |
| **效能優化** | 難以針對讀寫分別優化 | 可使用不同資料源 |
| **測試** | 測試案例混雜 | 測試案例清晰分類 |
| **API文檔** | Swagger分類困難 | Swagger自然分組 |

**✅ 推薦策略: Query與Command拆分式**

**推薦理由:**

1. **符合CQRS架構精神**
   - 本專案已採用CQRS架構 (參見5.2.3節)
   - Controller拆分是CQRS的自然延伸
   - 與現有的`CommandBaseController`和`QueryBaseController`基類配合

2. **配合Service Factory機制**
   - 現有的`CommandApiServiceFactory`和`QueryApiServiceFactory`分離設計
   - 避免Factory查找時的混淆

3. **利於讀寫分離資料庫**
   - Query Controller可配置讀取從庫或Elasticsearch
   - Command Controller固定寫入主庫

4. **Swagger文檔自動分組**
   - `@Tag` 自然區分Command與Query操作
   - API文檔更清晰易讀

5. **微服務粒度控制**
   - 未來可將Query端獨立部署成微服務
   - 支援更細粒度的水平擴展

**拆分規則:**

```
同一Resource的API依操作類型拆分:
├── HR{DD}{Screen}CmdController - POST, PUT, PATCH, DELETE
└── HR{DD}{Screen}QryController - GET
```

```java
/**
 * 員工到職 Domain Service
 * 負責處理員工到職的完整業務邏輯
 */
@Service
public class EmployeeOnboardingDomainService {
    
    private final IEmployeeRepository employeeRepository;
    private final IDepartmentRepository departmentRepository;
    private final DomainEventPublisher eventPublisher;
    
    /**
     * 執行員工到職流程
     * - 驗證部門存在
     * - 驗證主管存在
     * - 建立員工聚合根
     * - 發布EmployeeCreatedEvent
     */
    public Employee executeOnboarding(OnboardingCommand command) {
        // 1. 驗證部門
        Department department = departmentRepository
            .findById(command.getDepartmentId())
            .orElseThrow(() -> new DomainException("部門不存在"));
        
        // 2. 驗證主管
        if (command.getManagerId() != null) {
            employeeRepository.findById(command.getManagerId())
                .orElseThrow(() -> new DomainException("主管不存在"));
        }
        
        // 3. 建立員工 (透過聚合根工廠方法)
        Employee employee = Employee.onboard(command);
        
        // 4. 發布領域事件
        eventPublisher.publish(new EmployeeCreatedEvent(
            employee.getId(),
            employee.getEmployeeNumber(),
            employee.getCompanyEmail(),
            employee.getDepartmentId(),
            employee.getHireDate()
        ));
        
        return employee;
    }
}

/**
 * 帳號鎖定 Domain Service
 * 負責處理登入失敗後的帳號鎖定邏輯
 */
@Service
public class AccountLockingDomainService {
    
    private static final int MAX_FAILED_ATTEMPTS = 5;
    private static final int LOCK_DURATION_MINUTES = 30;
    
    /**
     * 記錄登入失敗並檢查是否需要鎖定
     */
    public boolean recordFailureAndCheckLock(User user) {
        user.incrementFailedAttempts();
        
        if (user.getFailedLoginAttempts() >= MAX_FAILED_ATTEMPTS) {
            user.lock(LocalDateTime.now().plusMinutes(LOCK_DURATION_MINUTES));
            return true;
        }
        
        return false;
    }
    
    /**
     * 檢查帳號是否已解鎖
     */
    public boolean checkAndUnlock(User user) {
        if (user.isLocked() && user.getLockUntil().isBefore(LocalDateTime.now())) {
            user.unlock();
            return true;
        }
        return false;
    }
}
```

#### 5.5.8 Business Pipeline 命名規範

當 Service 包含複雜業務邏輯（多步驟、多資料來源）時，使用 Pipeline 模式組織代碼。

> 詳細架構請參閱 [framework/architecture/03_Business_Pipeline.md](../framework/architecture/03_Business_Pipeline.md)

**Pipeline 組件命名:**

| 組件類型 | 命名格式 | 範例 | 說明 |
|:---|:---|:---|:---|
| **Context** | `{UseCase}Context` | `SalaryContext`, `CreateUserContext` | 數據載體，傳遞輸入/中間/輸出數據 |
| **Infrastructure Task** | `Load{Entity}Task` | `LoadEmployeeTask`, `LoadAttendanceTask` | 負責資料存取 |
| **Domain Task** | `{動詞}{業務}Task` | `CalculateTaxTask`, `ValidateEmailTask` | 負責業務計算 |
| **Integration Task** | `{動詞}{Service}Task` | `SendNotificationTask`, `CallPaymentTask` | 負責外部服務調用 |

**目錄結構範例:**

```
application/service/{domain}/
├── {動詞}{名詞}ServiceImpl.java      # Service (Pipeline 編排)
├── context/
│   └── {UseCase}Context.java         # 數據載體
└── task/
    ├── Load{Entity}Task.java         # Infrastructure Task
    ├── {動詞}{業務}Task.java          # Domain Task
    └── {動詞}{Service}Task.java       # Integration Task
```

**Service 與 Pipeline 範例:**

```java
@Service("calculateMonthlySalaryServiceImpl")
@Transactional
public class CalculateMonthlySalaryServiceImpl
        extends BaseCommandService<CalculateSalaryRequest, SalaryResponse> {

    private final LoadEmployeeTask loadEmployeeTask;
    private final LoadAttendanceTask loadAttendanceTask;
    private final CalculateTaxTask calculateTaxTask;

    @Override
    protected SalaryResponse doExecute(CalculateSalaryRequest req, JWTModel user) {
        // 1. 建立 Context
        SalaryContext ctx = new SalaryContext(req.getEmployeeId(), req.getMonth());

        // 2. 執行 Pipeline
        BusinessPipeline.start(ctx)
            .next(loadEmployeeTask)        // 步驟一：載入員工
            .next(loadAttendanceTask)      // 步驟二：載入出勤
            .next(calculateTaxTask)        // 步驟三：計算稅額
            .execute();

        // 3. 轉換回應
        return mapper.toResponse(ctx.getResult());
    }
}
```
