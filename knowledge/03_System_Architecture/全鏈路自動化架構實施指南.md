# 🚀 2025 全鏈路自動化架構實施指南 (The Architect's Blueprint)

## 0. 核心設計哲學：規格即實作 (Spec as Implementation)

架構師的目標是建立一個「法治」系統：

1. **資料層 (Data)**：透過 `QueryGroup` 讓查詢意圖化，消除重複的 CRUD 代碼。
2. **驗證層 (Verification)**：以 Markdown 作為合約，讓測試自動化且具備業務語義。
3. **邏輯層 (Logic)**：利用 `Pipeline` 模式，將複雜業務拆解為可觀測、可測試的細小零件。

---

## 1. 資料層：宣告式查詢引擎 (The Data Engine)

利用泛型與意圖物件，讓工程師專注於「要什麼」，而非「怎麼查」。

### 核心組件：`QueryGroup`

```java
public class QueryGroup {
    private List<Filter> filters; // Key, Operator, Value
    private List<String> sorts;
    private PageInfo page;
}

// Repository 實作：一次定義，終身免修
public abstract class BaseRepository<T> {
    public Page<T> findByQueryGroup(QueryGroup qg) {
        // 動態翻譯成 SQL / Criteria / Specification
        return queryEngine.translate(qg);
    }
}

```

> **價值**：降低 80% 的 Repository 開發成本，AI 生成代碼時不再需要長串的實作細節。

---

## 2. 驗證層：合約驅動測試 (The Contract Judge)

將 SA 的需求文件轉化為執行斷言，解決「球員兼裁判」的溝通斷層。

### 協作 SOP：

1. **SA 撰寫 Markdown**：定義場景 ID 與必須包含的過濾條件（如 `status = 'ACTIVE'`）。
2. **工程師攔截意圖**：測試時不比對資料庫結果，只比對產出的 `QueryGroup` 快照。

### 自動化判官實作：

```java
public void assertContract(QueryGroup actual, String markdownSpec, String scenarioId) {
    List<String> expectedRules = parseMarkdown(markdownSpec, scenarioId);
    expectedRules.forEach(rule -> {
        if (!actual.matches(rule)) throw new ContractViolationException(rule);
    });
}

```

## 3. 邏輯層：宣告式業務流水線 (The Business Pipeline)

針對「薪資計算」等複雜邏輯，將 Service 重構為透明的傳送帶，防止「接手即重寫」的悲劇。

### 結構模型：

```java
@Service
public class SalaryService {
    public SalaryContext calculate(Long empId) {
        return BusinessPipeline.start(new SalaryContext(empId))
            .next(fetchData)        // 步驟一：按需抓取 (Lazy Loading)
            .next(applyTaxRules)    // 步驟二：規則演算
            .nextIf(ctx -> ctx.isFullTime(), bonusTask) // 條件分支
            .execute();
    }
}

```

> **應對開發直覺**：這套模式雖然初看「死板」，但它提供了極佳的**除錯觀測點**。工程師可以隨時在步驟間插入 `logSnapshot()`，精確定位邏輯斷點。

## 6.架構整合


以下是將流水線方案套入現有規範的實施建議：

---

### 1. 職責層級的對接 (Integration Mapping)

將流水線 (Pipeline) 定位在你的 **Application Layer** 與 **Domain Layer** 之間：

* **Controller (Interface Layer)**: 保持不變，繼續使用 `execCommand`。
* **Service Impl (Application Layer)**: 扮演 **流水線的編排者 (Orchestrator)**。
* **Pipeline Tasks (Domain/Application Layer)**: 扮演 **邏輯的執行者 (Processors)**。

---

### 2. 實作範例：以規範中的 `CreateUserServiceImpl` 為例

根據你的規範 5.2 與 7.2，我們可以將 Service 改造如下：

```java
@Service("createUserServiceImpl")
@Transactional
public class CreateUserServiceImpl
        implements CommandApiService<CreateUserRequest, CreateUserResponse> {

    @Autowired private UserTasks userTasks; // 注入預定義的 Task 零件

    @Override
    public CreateUserResponse execCommand(CreateUserRequest req, JWTModel currentUser, String... args) {
        // 使用宣告式流水線取代原本的寫死邏輯
        UserContext context = BusinessPipeline.start(new UserContext(req, currentUser))
            .next(userTasks.validateEmailUnique())   // 驗證邏輯
            .next(userTasks.createNewUser())         // 領域物件創建
            .next(userTasks.saveToRepository())      // 持久化
            .next(userTasks.publishCreatedEvent())   // 領域事件派發 (規範 5.3)
            .execute();

        return new CreateUserResponse(context.getUserId());
    }
}

```

### 3. 如何解決「薪資計算」複雜度？

針對規範中的 `04-PAY` (薪資管理) Domain，當你面對派遣人員薪資計算時，你的目錄結構會演進為：

* **application/service/pay/**:
* `CalculateSalaryServiceImpl.java` (流水線編排)
* `task/` (存放細碎的計算零件)
* `FetchAttendanceTask.java`
* `CalculateOvertimeTask.java`
* `TaxDeductionTask.java`





**這對目前架構的優點：**

1. **符合 DDD 規範**: 這些 Task 可以進一步封裝成 `Domain Service`，保持 Application Service 的純粹。
2. **與 AOP 機制相容**: 你的 `ApiServiceAspect` 依然能運作，它只需要負責把請求送到 Service，而 Service 內部的 Pipeline 則保證了邏輯的透明度。

### 5. 結論：架構的終極進化

目前的架構已經解決了 **「怎麼找到正確的 Service (Where)」**，而流水線方案則解決了 **「Service 裡面怎麼寫 (How)」**。

* **Repository 泛型化**：解決資料存取成本。
* **動態 Service 派發**：解決 Controller 膨脹與依賴混亂。
* **宣告式流水線**：解決核心業務邏輯的不可維護性。
