## ⚠️ 仍存在的問題分析

### 問題類型一：合約佔位符未正確替換（8 個）

**影響場景：** IAM_QRY_001, 004, 101, 102, 103, 104

**錯誤訊息：**
```
缺失的過濾條件:
  ❌ tenant_id = {currentUserTenantId}
實際產出的過濾條件:
  ✓ tenant_id = 'T001'
```

**根因分析：**

測試程式使用兩種 API：
1. **舊 API：** `assertContract(queryGroup, contractSpec, scenarioId)` - 使用 setUp() 中預先載入的 `contractSpec` 字串
2. **新 API：** `loadContract(serviceName, scenarioId)` - 每次都重新從檔案載入

當前的佔位符替換在 `setUp()` 中：
```java
contractSpec = loadContractSpec("iam");
contractSpec = contractSpec.replace("{currentUserTenantId}", "T001");
```

但 Query 測試使用新 API：
```java
ContractSpec contract = loadContract("iam", "IAM_QRY_001");  // ← 重新載入，未替換！
verifyQueryContract(queryGroup, responseJson, contract);
```

**解決方案：**

方案一：在 `loadContract()` 後手動替換
```java
ContractSpec contract = loadContract("iam", "IAM_QRY_001");
// 替換 JSON 中的佔位符（需要修改 ContractSpec 支援此操作）
```

方案二：修改合約文件，不使用佔位符
```json
{"field": "tenant_id", "operator": "=", "value": "T001"}
```

方案三：修改 BaseContractTest.loadContract() 支援參數替換

**建議：** 採用方案二（修改合約文件），因為佔位符會增加複雜度

---

### 問題類型二：合約格式問題（2 個）

**影響場景：** AUTH_CMD_001, AUTH_CMD_003

**錯誤訊息：**
```
Unrecognized field "value" (class RequiredField),
not marked as ignorable (5 known properties: "type", "name", "precision", "format", "notNull"])
```

**根因分析：**

合約 JSON 中的 `requiredFields` 包含了 `value` 欄位：
```json
{"name": "tokenType", "type": "string", "value": "Bearer"}
```

但 `RequiredField` 類別不支援 `value` 屬性。

**解決方案：**

修改合約文件，移除 `value` 欄位或改用 `assertions`：
```json
// ❌ 錯誤
{"name": "tokenType", "type": "string", "value": "Bearer"}

// ✅ 正確
{"name": "tokenType", "type": "string", "notNull": true}
```

並在 `assertions` 中驗證值：
```json
"assertions": [
  {"field": "tokenType", "operator": "equals", "value": "Bearer"}
]
```

### 問題類型三：Command API 權限配置（5-6 個）

**影響場景：** IAM_CMD_002, 003, 004, 005, 006

**錯誤訊息：**
```
Status expected:<200> but was:<403>
```

**根因分析：**

雖然已在測試中加入權限：
```java
auths.add("user:write");
auths.add("role:write");
```

但測試仍回傳 403，可能原因：
1. Controller 的 `@PreAuthorize` 註解要求不同的權限格式
2. 或使用 `hasRole()` 而非 `hasAuthority()`
3. 或 Service 層有額外的權限檢查

**解決方案：**

檢查 Controller 的權限註解：
```bash
grep -r "@PreAuthorize" backend/hrms-iam/src/main/java/.../controller/
```

查看實際要求的權限格式並調整測試。

## 💡 關鍵學習與洞察

### 1. 合約測試的真正價值

**傳統觀點：**
> 「通過率 0% = 開發失敗」

**正確觀點：**
> 「通過率 0% = 發現了所有隱藏問題」

**證據：**
- 發現 8 個 Service 缺少軟刪除過濾
- 發現 6 個資料庫表格未建立
- 發現 2 個合約格式錯誤
- 發現 5-6 個權限配置問題

**價值：** 這些問題如果沒有合約測試，將在生產環境造成嚴重bug（例如：查詢到已刪除的資料）

---

### 2. QueryGroup 的重要性

**發現：** 直接使用 `findById()` 無法被合約測試驗證

**原則：** 所有 Query Service 都應使用 QueryGroup

**範例：**
```java
// ❌ 錯誤：無法驗證過濾條件
User user = repository.findById(userId);

// ✅ 正確：可被合約測試驗證
QueryGroup query = QueryBuilder.where()
    .eq("user_id", userId)
    .eq("is_deleted", false)
    .build();
User user = repository.findAll(query).stream().findFirst();
```

**好處：**
- 安全性過濾明確可見
- 可被自動化測試驗證
- 統一查詢模式

---

### 3. 測試基礎設施的關鍵性

**問題：** Command 測試需要真實資料庫進行快照比對

**解決方案：**
- Testcontainers（理想，需 Docker）
- H2 PostgreSQL 模式（實用，無需 Docker）

**效果：**
- Errors 從 6 → 0
- Command 測試從「完全無法執行」變成「可執行但驗證失敗」

---

### 4. 合約格式的一致性

**問題：** 合約使用佔位符和特殊欄位增加了複雜度

**建議：**
- 避免使用佔位符（或提供統一替換機制）
- 嚴格遵守 ContractSpec 的 JSON Schema
- 不支援的欄位應移到 `assertions` 中

---

### 5. 漸進式修正的有效性

**流程：**
1. P0：解除阻塞（資料庫） → Errors 歸零
2. P1：核心問題（軟刪除） → 安全性改善
3. P2：次要問題（權限） → 功能完整性

**效果：**
- 每個階段都有明確改善
- 問題逐步減少
- 進度可量化
