## ⚠️ 尚存問題分析

### 問題類型一：tenant_id 佔位符未替換（8 個）

**影響場景：** IAM_QRY_001, 004, 101, 102, 103, 104

**錯誤訊息：**
```
缺失的過濾條件:
  ❌ tenant_id = {currentUserTenantId}
實際產出的過濾條件:
  ✓ tenant_id = 'T001'
```

**問題分析：**
- 實作是正確的（有 `tenant_id = 'T001'` 過濾）
- 合約中使用 `{currentUserTenantId}` 佔位符
- 測試執行時沒有替換佔位符

**解決方案：**

在測試的 `setUp()` 中替換佔位符：

```java
@BeforeEach
void setUp() throws Exception {
    mockAdminUser = new JWTModel();
    mockAdminUser.setTenantId("T001");

    contractSpec = loadContractSpec("iam");
    // ← 需要替換佔位符
    contractSpec = contractSpec.replace("{currentUserTenantId}", "T001");
}
```

**影響：** 這是測試的小問題，不是實作問題

## 💡 關鍵學習

### 1. 合約測試的驗證機制

**發現：** 合約測試不僅驗證「功能可用」，更驗證「實作細節」

**證據：**
- `is_deleted = false` 必須明確出現在 QueryGroup 中
- 不能依賴資料庫視圖或其他隱式過濾
- 每個安全性過濾條件都會被驗證

### 2. QueryGroup 的重要性

**原則：** 所有 Query Service 都應使用 QueryGroup，而非直接 Repository 方法

**優點：**
- 可被合約測試驗證
- 安全性過濾集中管理
- 統一查詢模式

**修正範例：**
```java
// ❌ 錯誤：無法驗證過濾條件
User user = repository.findById(userId);

// ✅ 正確：可被合約測試驗證
QueryGroup query = QueryBuilder.start()
    .eq("user_id", userId)
    .eq("is_deleted", false)
    .build();
User user = repository.findAll(query).stream().findFirst();
```

### 3. 測試失敗不代表失敗

**觀點轉變：**
- ❌ 錯誤觀點：「測試通過率 0% = 開發失敗」
- ✅ 正確觀點：「測試通過率 0% = 找到所有問題」

**價值：**
- 合約測試發現了 19 個實作缺陷
- 每個缺陷都是潛在的安全性或業務邏輯問題
- 修正後的程式碼品質更高

### 4. 漸進式修正的重要性

**流程：**
1. P0：解除阻塞（資料庫設定） → Errors 從 6 → 0
2. P1：修正核心問題（軟刪除過濾） → 實際過濾條件改善
3. P2：修正次要問題（權限配置） → 預期提升通過率

**效果：**
- 每個階段都有明確的改善
- 問題逐步減少
- 進度可量化追蹤
