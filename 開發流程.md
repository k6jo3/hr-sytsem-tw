### 第一階段：定義契約與測試 (Contract & Test First)

這是確保「程式照需求跑」的關鍵步驟。

1. **解析 Markdown 規格書**：SA 定義好業務場景（如：派遣人員加班費計算規則）。
2. **撰寫失敗的測試 (RED)**：根據規範 8.2，建立測試案例並定義預期的 **QueryGroup** 或 **業務結果快照**。
* *目的*：先決定「正確的結果」長什麼樣子，這就是您的開發終點。



### 第二階段：領域建模 (Domain Modeling)

在動手寫 API 之前，先建立核心業務邏輯的基礎。

3. **建立 Domain Entity 與 Aggregate Root**：根據規範 5.3，定義純淨的 Java 物件（如 `User` 或 `SalaryResult`）。
4. **建立 Domain Service**：若邏輯涉及多個領域物件，則建立 `{業務事件}DomainService`。
* *核心原則*：這裡只處理「純業務規則」，不涉及資料庫存取。



### 第三階段：流水線拆解 (Context & Tasks)

將複雜的業務行為分解為可管理的零件。

5. **定義 Context**：建立該 API 專屬的 `Context` 物件，用來傳遞數據。
6. **實作 Tasks**：將業務邏輯拆解為細小的 `Task`。
* **Infrastructure Task**：負責資料存取（MyBatis 或 Querydsl）。
* **Domain Task**：負責調用剛才建立的 Domain Service 進行計算。



### 第四階段：API 組裝與實作 (Application Integration)

最後一步才是將零件組合成服務。

7. **建立 Service Impl**：建立 `{動詞}{名詞}ServiceImpl`，並明確指定 Bean 名稱。
8. **編排 Pipeline**：在 `execCommand` 中，使用 `BusinessPipeline` 照序連結所有 Task。
9. **實作 Controller**：建立 `HR{DD}{Screen}CmdController` 並繼承 `CommandBaseController`。
* *註*：此時 Controller 僅需調用 `execCommand(request, currentUser)` 即可完成實作。



### 第五階段：驗證與重構 (Green & Refactor)

10. **跑測試 (GREEN)**：確保剛才寫的代碼能通過第一階段定義的合約測試。
11. **重構 (REFACTOR)**：優化代碼結構，確保符合 **SOLID** 與**命名規範**。
