## 2. 三大設計原則

### 原則一：宣告優於程式碼

> 能用「宣告」的，就不要用「程式碼」

**實踐方式:**

| 場景 | 傳統做法 | 宣告式做法 |
|:---|:---|:---|
| 查詢條件 | 手寫 if-else 組裝 | `@QueryFilter` 註解 |
| DTO 映射 | 手寫 getter/setter | MapStruct 自動映射 |
| 驗證規則 | Service 內 if 判斷 | `@Valid` + Bean Validation |
| API 文檔 | 另外維護文檔 | Swagger 註解自動生成 |

**效益:**
- 減少重複程式碼
- 降低人為錯誤
- 提升可讀性
- 便於維護

### 原則二：結構化輸出

> 能回傳「結構化物件」的，就不要回傳「基本型別」

**實踐方式:**

```java
// 查詢結果：回傳 QueryGroup 而非 SQL
QueryGroup group = QueryBuilder.where()
    .fromDto(request)
    .build();

// 計算結果：回傳 CalculationResult 而非數字
CalculationResult<BigDecimal> result = employee.calculateBonus(policy);

// API 回應：回傳完整 Response 物件而非單一值
UserDetailResponse response = mapper.toResponse(user);
```

**效益:**
- 便於快照測試
- 可追蹤計算過程
- 易於除錯
- 自動化驗證

### 原則三：Service 只編排

> Service 只做「編排（Orchestration）」，不做「決策（Decision）」

**正確的職責分配:**

| 層級 | 職責 | 範例 |
|:---|:---|:---|
| **Domain** | 業務規則、計算邏輯 | `Order.calculateTotal()` |
| **Service** | 流程編排、外部整合 | 載入資料 → 呼叫 Domain → 儲存 → 發事件 |
| **Controller** | HTTP 處理、參數驗證 | 接收請求 → 呼叫 Service → 回傳結果 |

**判斷標準:**
- 如果 Service 內有複雜的 `if-else`，代表業務邏輯洩漏
- 如果 Service 需要大量 Mock 才能測試，代表職責不清

## 3. 架構驅動測試

### 3.1 傳統測試 vs 架構驅動測試

| 面向 | 傳統測試 | 架構驅動測試 |
|:---|:---|:---|
| 測試對象 | 程式碼怎麼跑 (How) | 產出的數據合約 (What) |
| Mock 使用 | 到處都是 Mock | 幾乎不 Mock，只攔截結果物件 |
| DB 依賴 | 沒資料庫就沒法測 | 核心邏輯完全脫離資料庫 |
| 維護反饋 | 「改 A 壞 B」 | 「精準報警」 |
| 測試代碼量 | 比業務代碼還長 | 3-5 行搞定 |

### 3.2 測試即文檔

快照 JSON 就是最精準的系統說明書：

```json
// snapshots/query/employee_search_by_dept.json
{
  "junction": "AND",
  "conditions": [
    { "f": "department.name", "op": "EQ", "v": "研發部" },
    { "f": "status", "op": "EQ", "v": "ACTIVE" }
  ]
}
```

新進同事不用看 SQL，看這些 JSON 就能理解業務規則。

## 5. 架構品質指標

### 5.1 健康度檢測

透過測試的寫法來自我檢測架構品質：

| 指標 | 差 (Bad) | 好 (Good) |
|:---|:---|:---|
| **Setup 成本** | Mock 5+ 物件、準備 10+ 資料表 | Mock 0-1 物件，無需資料庫 |
| **維護成本** | 改 1 欄位壞 50 測試 | 只有相關測試失敗 |
| **執行速度** | 改 1 行等 3 分鐘 | 10 秒內跑完 |

### 5.2 反模式警告

當你發現以下情況，代表架構需要重構：

- Service 內有超過 3 層的 `if-else`
- 測試需要 Mock 超過 3 個依賴
- 無法用快照測試驗證業務邏輯
- 改動一處影響多處不相關的測試

## 6. 架構師守則

> 「架構師的工作不是寫更多的代碼，而是**消除不必要的代碼**。如果你發現工程師開始在 Service 裡寫複雜的 `if-else`，那是架構的失職，請立刻重構底層引擎，將邏輯收回架構中。」

### 6.1 日常檢視

1. **每週 Code Review** - 檢查是否有業務邏輯洩漏到 Service
2. **測試覆蓋率** - 快照測試應佔 80% 以上
3. **新人上手時間** - 理想情況 1 週內可獨立開發

### 6.2 重構時機

- 新功能難以用現有模式實作
- 測試變得難以撰寫
- 重複的樣板代碼出現
