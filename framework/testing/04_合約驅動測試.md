## 一、 系統運作流程圖 (Process Flow)

本架構的核心在於將「需求」直接轉化為「測試斷言」，減少人為轉譯損失。

1. **規格定義 (Define):** SA 在 Markdown 定義業務場景與必須包含的過濾條件 (位於根目錄 `/contracts`)。
2. **實作 (Implement):** 工程師根據需求開發 Controller 與 Service。
3. **攔截 (Intercept):** 測試執行時，Mock 掉數據庫層，攔截 Service 產出的 `QueryGroup` 意圖物件。
4. **解析 (Parse):** `ContractEngine` 解析 Markdown 表格內容。
5. **斷言 (Assert):** 比對 `QueryGroup` 裡的過濾器是否完全滿足 Markdown 的合約要求。

## 二、 Markdown 規格書實例 (`contracts/iam_contracts_v2.md`)

這是 SA 的工作台，用來鎖定業務邏輯與權限規則。合約文件統一放置於專案根目錄的 `/contracts` 資料夾下。

### 2.1 雙層結構設計

**重要：** 新版合約文件採用**雙層結構**設計，兼顧測試自動化與業務理解：

1. **機器可讀層 (Machine-Readable Layer)** - Markdown 表格
   - 用於 `MarkdownContractEngine` 自動解析與驗證
   - 定義 API 端點、輸入參數、必須包含的查詢條件
   - 支援自動化測試執行

2. **人類可讀層 (Human-Readable Layer)** - 詳細業務描述
   - 業務規則說明
   - 權限檢查邏輯
   - 異常處理規範
   - 測試範例程式碼

### 2.2 查詢操作合約範例 (Query Contract)

#### 機器可讀表格 (用於自動化測試)

```markdown
#### 2.1.2 機器可讀合約表格 (For Test Automation)

| 場景 ID | 測試描述 | 模擬角色 | API 端點 | 輸入 (Request) | 必須包含的過濾條件 (Required Filters) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| IAM_QRY_001 | 查詢啟用中的使用者 | ADMIN | `GET /api/v1/users` | `{"status":"ACTIVE"}` | `status = 'ACTIVE'`, `tenant_id = '{currentUserTenantId}'` |
| IAM_QRY_002 | 依帳號模糊查詢 | ADMIN | `GET /api/v1/users` | `{"keyword":"admin"}` | `username LIKE '%admin%'`, `status != 'DELETED'` |
| IAM_QRY_003 | 多租戶權限隔離 | USER | `GET /api/v1/users` | `{}` | `tenant_id = '{currentUserTenantId}'`, `status != 'DELETED'` |
```

**說明：**
- `{currentUserTenantId}` 表示當前使用者的租戶 ID，測試時會自動替換
- `status != 'DELETED'` 用於過濾已刪除的資料 (軟刪除)
- 表格中新增 `API 端點` 欄位，明確指定測試的 API

#### 詳細業務描述 (提供給開發者理解)

**場景 IAM_QRY_001: 查詢啟用中的使用者**

- **業務規則:**
  1. 僅返回狀態為 ACTIVE 的使用者
  2. 自動套用多租戶隔離過濾 (tenant_id)
  3. 管理員角色可查詢所有租戶資料

- **權限檢查:**
  - 需要 `USER_VIEW` 權限
  - ADMIN 角色不受租戶限制

- **測試範例:**
```java
@Test
@DisplayName("IAM_QRY_001: 查詢啟用中的使用者")
void searchActiveUsers_ShouldFilterByStatusAndTenant() throws Exception {
    String contractSpec = loadContractSpec("iam");
    GetUserListRequest request = GetUserListRequest.builder()
        .status("ACTIVE")
        .build();

    verifyApiContract("/api/v1/users", request, contractSpec, "IAM_QRY_001");
}
```

### 2.3 命令操作合約範例 (Command Contract)

命令操作不使用 `QueryGroup` 攔截，而是驗證**業務規則執行**和**事件發布**。

#### 機器可讀表格

```markdown
| 場景 ID | 測試描述 | 模擬角色 | API 端點 | 輸入 (Request) | 必須檢查的業務規則 | 預期發布的事件 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| IAM_CMD_001 | 建立使用者 | ADMIN | `POST /api/v1/users` | `{"username":"john","email":"john@example.com"}` | 帳號唯一性檢查, Email 格式驗證 | `UserCreatedEvent` |
| IAM_CMD_002 | 鎖定使用者 | ADMIN | `PUT /api/v1/users/{id}/lock` | `{"reason":"違規操作"}` | 狀態轉換檢查 | `UserLockedEvent` |
```

#### 詳細業務描述

**場景 IAM_CMD_001: 建立使用者**

- **業務規則:**
  1. 帳號必須唯一 (username unique constraint)
  2. Email 格式必須有效
  3. 預設狀態為 ACTIVE
  4. 自動分配當前使用者的租戶 ID

- **Domain Event:**
  - 成功建立後發布 `UserCreatedEvent`
  - Event 包含: userId, username, email, tenantId

- **測試範例:**
```java
@Test
@DisplayName("IAM_CMD_001: 建立使用者")
void createUser_ShouldPublishUserCreatedEvent() throws Exception {
    CreateUserRequest request = CreateUserRequest.builder()
        .username("john")
        .email("john@example.com")
        .build();

    mockMvc.perform(post("/api/v1/users")
            .contentType(MediaType.APPLICATION_JSON)
            .content(objectMapper.writeValueAsString(request)))
        .andExpect(status().isOk());

    // 驗證 Domain Event 發布
    verify(eventPublisher).publish(argThat(event ->
        event instanceof UserCreatedEvent &&
        ((UserCreatedEvent) event).getUsername().equals("john")
    ));
}
```

## 三、 Markdown 解析與斷言引擎實作

這是架構師提供的核心組件，用於自動化執行 SA 的判決。

```java
import java.util.*;
import java.util.regex.*;
import java.util.stream.Collectors;

public class MarkdownContractEngine {

    // 支援語義解析：Key [運算子] Value (例如: status = 'ACTIVE')
    private static final Pattern CRITERIA_PATTERN = Pattern.compile("(\\w+)\\s*([=><!\\sLIKE]+)\\s*'?([^']+)'?");

    /**
     * 核心斷言：驗證 QueryGroup 是否符合 Markdown 定義
     * @param actualQuery 程式碼產出的查詢物件
     * @param markdownTable 來自 SA 的規格文本
     * @param scenarioId 指定驗證哪一個場景
     */
    public void assertContract(QueryGroup actualQuery, String markdownTable, String scenarioId) {
        List<String> requiredFilters = parseFiltersFromTable(markdownTable, scenarioId);

        for (String criteria : requiredFilters) {
            boolean isMatched = actualQuery.getFilters().stream()
                    .anyMatch(filter -> verifyFilterMatch(filter, criteria));

            if (!isMatched) {
                throw new AssertionError(String.format(
                    "❌ 合約違規 [%s]：預期應包含條件 [%s]，但實際產出的查詢條件中未找到。",
                    scenarioId, criteria));
            }
        }
        System.out.println("✅ 場景 [" + scenarioId + "] 驗證通過，符合業務合約。");
    }

    private boolean verifyFilterMatch(Filter filter, String criteria) {
        Matcher m = CRITERIA_PATTERN.matcher(criteria);
        if (m.find()) {
            String key = m.group(1);
            String op = m.group(2).trim();
            String val = m.group(3);
            // 比對 Key 與 Value (Operator 可進一步映射 Enum)
            return filter.getKey().equalsIgnoreCase(key) &&
                   filter.getValue().toString().equalsIgnoreCase(val);
        }
        return false;
    }

    private List<String> parseFiltersFromTable(String table, String scenarioId) {
        return table.lines()
                .filter(line -> line.contains(scenarioId))
                .map(line -> line.split("\\|")[6].trim()) // 取第 6 欄：必須包含的條件（新增 API 端點後變為第 6 欄）
                .flatMap(s -> Arrays.stream(s.split(",")))
                .map(String::trim)
                .collect(Collectors.toList());
    }
}
```

**重要變更說明:**
- 新版合約表格新增了 `API 端點` 欄位，因此 `parseFiltersFromTable` 方法中的欄位索引從第 5 欄改為第 6 欄
- 支援動態值替換，如 `{currentUserTenantId}` 會在測試執行時自動替換為實際值

## 四、 測試執行範例 (Flow Integration)

開發者只需調用基類方法，即可完成全鏈路驗證。

### 4.1 查詢操作測試

```java
class UserContractTest extends BaseContractTest {

    @Test
    void searchActiveUsers_ShouldMatchContract() throws Exception {
        // 1. 載入合約規格 (從 contracts/ 目錄)
        String contract = loadContractSpec("iam"); // 對應 contracts/iam_contracts_v2.md

        // 2. 準備請求
        GetUserListRequest req = new GetUserListRequest();
        req.setStatus("ACTIVE");

        // 3. 驗證合約 (場景 IAM_QRY_001)
        verifyApiContract("/api/v1/users", req, contract, "IAM_QRY_001");
    }
}
```

### 4.2 命令操作測試

```java
class UserCommandContractTest extends BaseContractTest {

    @Test
    void createUser_ShouldMatchContract() throws Exception {
        // 1. 載入合約規格
        String contract = loadContractSpec("iam");

        // 2. 準備請求
        CreateUserRequest req = CreateUserRequest.builder()
            .username("john")
            .email("john@example.com")
            .build();

        // 3. 驗證命令合約 (場景 IAM_CMD_001)
```

## 五、 本架構之核心價值

1. **非對稱驗證 (Asymmetric Verification):** 工程師不再是自己測試自己，而是由 SA 定義的 Markdown 作為「外部法規」。
2. **業務可視化:** 所有的過濾邏輯（如刪除標記、發佈狀態、權限過濾）都顯性化地寫在 Markdown 中，而非隱藏在深層代碼。
3. **溝通革命:** PM/SA 看到測試失敗時，會知道是「合約未達成」，而不只是「工程師有 Bug」。
4. **零 Mock 負擔:** 透過 `QueryGroup` 攔截技術，測試無需啟動資料庫，執行速度極快。
5. **雙層結構:** 同時滿足測試自動化（機器可讀）和業務理解（人類可讀）的需求。

## 六、 合約文件維護指南

### 6.1 何時更新合約

合約文件應該由 SA 維護，並在以下情況更新：

1. **新增功能:** 新增 API 端點或業務場景
2. **業務規則變更:** 修改過濾條件、權限檢查邏輯
3. **Domain Model 變更:** Entity 欄位新增/修改/刪除
4. **事件定義變更:** 新增或修改 Domain Events

### 6.2 合約變更流程

```
1. SA 更新合約文件
   ↓
2. 提交 PR (只修改 contracts/*.md)
   ↓
3. 測試執行 → 失敗 (預期行為)
   ↓
4. 開發者根據新合約修改實作
   ↓
5. 測試通過 → 合併 PR
```

### 6.3 注意事項

1. **欄位存在性檢查:** 合約中定義的欄位必須在 Entity/DTO 中實際存在
2. **API 端點一致性:** 合約中的 API 端點必須與實際 Controller 路徑一致
3. **角色權限對應:** 模擬角色必須與系統實際的角色定義一致
4. **事件命名規範:** Domain Events 命名遵循 `{Aggregate}{PastTense}Event` 格式
