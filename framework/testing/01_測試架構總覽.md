# 測試架構總覽

**版本:** 1.0
**目的:** 建立低成本、高可讀性、易維護的自動化測試體系

---

## 1. 測試金字塔

```
                    ┌─────────┐
                    │   E2E   │  ← 少量：關鍵業務流程
                   ┌┴─────────┴┐
                   │Integration│  ← 中量：API 契約測試
                  ┌┴───────────┴┐
                  │    Unit     │  ← 大量：Domain + 快照測試
                 └──────────────┘
```

## 2. 各層測試職責

| 層級 | 測試對象 | 方法 | 執行頻率 |
|:---|:---|:---|:---|
| **Unit** | Domain Entity 計算邏輯 | 快照比對 | 每次提交 |
| **Unit** | QueryGroup 組裝邏輯 | 快照比對 | 每次提交 |
| **Integration** | API → Service 契約 | MockMvc + 快照 | 每次提交 |
| **Integration** | QueryEngine → DB | H2 + 固定資料集 | 每日/PR |
| **E2E** | 關鍵業務流程 | Testcontainers | 發版前 |

## 3. 測試設計原則

### 3.1 傳統 vs 架構驅動測試

| 面向 | 傳統測試 | 架構驅動測試 |
|:---|:---|:---|
| 測試對象 | 程式碼怎麼跑 (How) | 產出的數據合約 (What) |
| Mock 使用 | 到處都是 Mock | 幾乎不 Mock，只攔截結果物件 |
| DB 依賴 | 沒資料庫就沒法測 | 核心邏輯完全脫離資料庫 |
| 維護反饋 | 「改 A 壞 B」 | 「精準報警」 |
| 測試代碼量 | 比業務代碼還長 | 3-5 行搞定 |

### 3.2 核心原則

1. **測試即文檔** - 快照 JSON 就是系統說明書
2. **零 Mock** - 透過結構化物件攔截
3. **快速執行** - 0.1 秒級別驗證
4. **精準報警** - 只有相關測試失敗

## 4. 三階測試法

### 4.1 概述

```
┌─────────────────────────────────────────────────────────────────┐
│                    三階測試法                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  第一階：引擎契約測試 (Engine Contract Test)                     │
│  ─────────────────────────────────────────                      │
│  對象：QueryEngine                                              │
│  做法：H2 固定資料集，測試所有 Operator                          │
│  目的：確保「翻譯機」不會壞，後續不需跑 DB                        │
│                                                                 │
│  第二階：業務組裝測試 (Business Assembly Test)                   │
│  ─────────────────────────────────────────                      │
│  對象：Application Service                                      │
│  做法：攔截 QueryGroup，驗證條件組裝是否正確                     │
│  工具：快照比對 或 DSL 斷言                                      │
│                                                                 │
│  第三階：合約驅動測試 (Contract-Driven Test)                     │
│  ─────────────────────────────────────────                      │
│  對象：API → QueryGroup 完整流程                                │
│  做法：解析 Markdown 合約，驗證是否符合 SA 規格                  │
│  目的：確保程式完全符合業務需求                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 第一階：引擎契約測試

一次性驗證 QueryEngine 的所有操作符：

```java
@SpringBootTest
class QueryEngineContractTest {

    @ParameterizedTest
    @MethodSource("operatorTestCases")
    void operator_ShouldProduceCorrectResult(Operator op, Object value, int expectedCount) {
        QueryGroup query = QueryBuilder.where()
            .and("status", op, value)
            .build();

        List<Entity> result = repository.findAll(query);

        assertThat(result).hasSize(expectedCount);
    }

    static Stream<Arguments> operatorTestCases() {
        return Stream.of(
            Arguments.of(Operator.EQ, "ACTIVE", 10),
            Arguments.of(Operator.NE, "ACTIVE", 5),
            Arguments.of(Operator.LIKE, "John%", 3),
            Arguments.of(Operator.IN, List.of("A", "B"), 7)
        );
    }
}
```

### 4.3 第二階：業務組裝測試

驗證 Service 組裝的 QueryGroup 是否符合業務意圖：

```java
class EmployeeQueryServiceTest extends BaseServiceTest {

    @Test
    void searchEmployee_ShouldBuildCorrectQuery() {
        GetEmployeeListRequest req = new GetEmployeeListRequest();
        req.setDepartmentName("研發部");
        req.setStatus("ACTIVE");

        service.getResponse(req, mockUser);

        // 快照比對
        FluentAssert.assertMatchesSnapshot(
            "employee_search_by_dept.json",
            service.getLastQueryGroup()
        );
    }
}
```

### 4.4 第三階：合約驅動測試

驗證 API 到 QueryGroup 的完整流程符合 SA 合約：

```java
class EmployeeContractTest extends BaseContractTest {

    @Test
    void searchEmployee_ShouldMatchContract() throws Exception {
        String contract = loadContractSpec("employee");

        EmployeeSearchRequest req = new EmployeeSearchRequest();
        req.setStatus("ACTIVE");

        verifyApiContract("/api/v1/employees/search", req, contract, "EMP-001");
    }
}
```

## 5. 快照測試

### 5.1 流程

```
開發階段 (Learn Mode)
─────────────────────
1. 開發 Service 邏輯
2. 執行測試 (加上 -DupdateSnapshots=true)
3. 系統自動將結果存為 JSON 快照
4. Review 快照內容，確認正確
5. 提交快照到 Git

驗證階段 (Strict Mode)
─────────────────────
1. CI/CD 執行測試
2. 比對產出與快照
3. 不一致則報錯，顯示 Diff
4. 確認為預期變更 → 更新快照
5. 確認為 Bug → 修正程式碼
```

### 5.2 快照範例

**QueryGroup 快照:**

```json
{
  "junction": "AND",
  "conditions": [
    { "f": "department.name", "op": "EQ", "v": "研發部" },
    { "f": "status", "op": "EQ", "v": "ACTIVE" }
  ]
}
```

**計算結果快照:**

```json
{
  "finalValue": 50000,
  "appliedRules": [
    "BASE_SALARY_RATE: 月薪 x 1.5",
    "PERFORMANCE_BONUS: 績效加成 10%"
  ],
  "breakdown": {
    "baseSalary": 40000,
    "performanceBonus": 10000
  }
}
```

### 5.3 執行命令

```bash
# 驗證模式 (CI/CD)
mvn test

# 學習模式 (更新快照)
mvn test -DupdateSnapshots=true
```

## 7. 測試基類

### 7.1 基類繼承關係

```
BaseTest (通用配置)
    │
    ├── BaseUnitTest (純單元測試)
    │       │
    │       ├── BaseDomainTest (Domain Entity 測試)
    │       └── BaseServiceTest (Service 測試)
    │
    └── BaseIntegrationTest (整合測試)
            │
            ├── BaseApiContractTest (API 契約)
            └── BaseContractTest (合約驅動測試)
```

### 7.2 使用範例

```java
// Service 測試
class GetEmployeeListServiceTest extends BaseServiceTest {

    @Test
    void searchByDepartment_ShouldMatchSnapshot() {
        GetEmployeeListRequest req = new GetEmployeeListRequest();
        req.setDepartmentName("研發部");

        verifyQuery(
            () -> service.getResponse(req, mockUser),
            "employee_search_by_dept"
        );
    }
}

// 合約測試
class EmployeeContractTest extends BaseContractTest {

    @Test
    void searchEmployee_ShouldMatchContract() throws Exception {
        String contract = loadContractSpec("employee");

        verifyApiContract(
            "/api/v1/employees/search",
            new EmployeeSearchRequest(),
            contract,
            "EMP-001"
        );
    }
}
```

## 8. CI/CD 整合

```yaml
# .github/workflows/test.yml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Run Tests (Strict Mode)
        run: mvn test
        # 不加 -DupdateSnapshots，任何不一致都會失敗

      - name: Upload Snapshot Diff
        if: failure()
        uses: actions/upload-artifact@v3
        with:
          name: snapshot-diff
          path: target/snapshot-diff/
```
